### Mutation experiments (mutation-proof bar)
Each interactional assertion is paired with the production-code mutation that would break it:

1. **AC1 Back-disabled mutation** (`packages/ui/src/Stepper.tsx:43`): if the implementer removes `if (isBackDisabled) return;` from `handleBack`, the click would advance to step -1 (or wrap), changing the active tab. The test reads `aria-current="step"` BEFORE the click, dispatches the click, and asserts the current tab is unchanged AFTER. Removing the guard breaks the test.
2. **AC1 Next-validation mutation**: if the wizard's `hasErrors` wiring is removed (Stepper called without `hasErrors={‚Ä¶}` reflecting the active step's RHF validity), the Next button's `aria-disabled` would not be `'true'` on initial render. The assertion `toHaveAttribute('aria-disabled', 'true')` fails.
3. **AC1 ESC mutation**: if the P1 story passes `dismissible={false}` instead of the default `true`, Radix's `onEscapeKeyDown` is preventDefaulted in Modal.tsx and `data-state="open"` remains; the `waitFor(() => stillOpen toBeNull)` times out.
4. **AC2 loading-state mutation**: if the P10 implementer collapses the state machine to a binary idle/result (skipping loading), the `[data-testid="dry-run-state-loading"]` element never appears; `waitFor` times out.
5. **AC2 error-state mutation**: if no `ErrorState` story export exists, `expect(ErrorStory).toBeTruthy()` fails before the render.
6. **AC2 Summary-on-right mutation**: if Summary is placed on the left instead of the right, `right!.querySelector('dl')` returns null.
7. **AC3 a11y mutation**: any axe-core serious/critical violation (missing label, broken aria-labelledby, contrast issue introduced via inline styles) fails `toHaveNoViolations`.

### Deviation evaluations

**1. `useSanitiseRadixIds` story hook (P1)**
Decision: **ACCEPT story-only.** The hook compensates for two distinct issues: (a) Radix Tabs generates `id`s with colons (`radix-:r55:-trigger-0`), which axe-core's `aria-valid-attr-value` flags in ARIA refs; (b) Stepper renders `Tabs.List` but no `Tabs.Content`, so generated `aria-controls` references dangle. These are a Radix test-environment quirk, not a Stepper production bug ‚Äî real browsers parse colon-ids correctly and `aria-controls` pointing to a panel that renders lazily is a valid progressive-disclosure pattern. Providing `Tabs.Content` slots in a story-only wizard would require stubbing all 8 step panels, adding complexity with no production value. The hook is scoped to `P1-Wizard.stories.tsx` and does not modify any primitive. Carry-forward T-062: document the Radix colon-id axe-core quirk in `packages/ui/TESTING.md` so future pattern authors know to apply the same hook or suppress the specific rule in `assertModalA11y`.

**2. `<label>` wrap of ReasonInput in P5/P9**
Decision: **ACCEPT implicit wrap.** The implicit `<label><span>‚Ä¶</span><ReasonInput/></label>` pattern is HTML5-valid and the axe `label-title-only` rule (which fires when a textarea has only `title`/`placeholder` as its accessible name) is correctly silenced. `ReasonInput`'s public contract is a controlled `name`/`minLength`/`placeholder` surface; adding `aria-label` forwarding would require touching the primitive (out of T-031 scope) and would be a lower-fidelity fix than a visible label. The implicit wrap is the right short-term solution. Carry-forward T-063: add `aria-label` / `forwardRef` to `ReasonInput` so callers can avoid the wrapper when a visible label is not desired.

**3. P10 ErrorState initial-render (no state-transition test)**
Decision: **ACCEPT as scoped.** AC2 requires testing "loading/result/error states" ‚Äî all three are exercised: `loading` and `result` via the click-driven transition test; `error` via the `ErrorState` story's initial render. The test comment explicitly acknowledges that presence at first render satisfies the structural mutation guard ("if no `ErrorState` export exists, `expect(ErrorStory).toBeTruthy()` fails"). A full click-to-error transition test would add confidence but is not required by the AC text. Carry-forward T-064: add a `shouldFail=true` click-driven transition test to patterns.test.tsx under AC2.

**4. P1 padded 3‚Üí8 steps**
Decision: **PASS-with-note.** T-031.json AC1 explicitly specifies 8 steps. The prototype (`email_template_edit_modal`) only ships 3 ‚Äî the implementer correctly padded to 8 with placeholder labels (Metadata, Recipients, Subject, Body, Variables, Schedule, Preview, Review) as directed in the RED notes. The step content beyond step 0 is a placeholder paragraph, which is acceptable for a pattern template. No carry-forward required ‚Äî labels are documented as placeholders.
