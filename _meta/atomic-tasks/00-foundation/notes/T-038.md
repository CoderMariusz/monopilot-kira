## Phase: REVIEW
Status: PASS
Reviewer agent type/model: claude-sonnet-4-6 (REVIEWER skill)
Findings:

**Concern 1 — FK to organizations(id) removed**
Acceptable with a known risk. The PRD says "UUID NOT NULL REF organizations(id)". GREEN removed it because tests insert random UUIDs without parent rows. The risk for canary upgrade orchestration is real but bounded: a stale `tenant_id` that no longer exists in `organizations` will sit silently in the table and could trigger ghost orchestration jobs. Mitigation path: T-039 (the orchestration Server Action) must guard on FK existence at the application layer before acting on any row, and a periodic cleanup job should prune orphaned rows. The decision is acceptable at this schema layer only if that application-layer contract is enforced in T-039. This is logged as a risk carry-forward, not a blocker for T-038.

**Concern 2 — `_` shim on the table object**
Confirmed necessary. At runtime under drizzle-orm 0.40.0, `pgTable(...)` produces an object with no `_` property (verified by direct node evaluation). The type declarations in `table.d.ts` still reference `_` as a type-level phantom but it is not present on the runtime value. The shim is the correct pragmatic fix. It is scoped to `tenant-migrations.ts` only, is clearly documented with a comment, and carries no downstream risk to drizzle-kit tooling (drizzle-kit reads the schema directory, not the compiled `_` at runtime). The code smell is accepted.

**Concern 3 — Migration filename 013-tenant-migrations.sql vs 0008_tenant_migrations.sql**
STATUS.md migration ordering lock (bottom of file) explicitly assigns 013 to T-038: `013 tenant-migrations (T-038)`. The RED notes also locked this. The task JSON's `scope_files` list of `0008_tenant_migrations.sql` is stale/incorrect — it predates the lock. The implementer correctly followed STATUS.md. Migration filename is correct.

**Concern 4 — Dual schema directories: packages/db/src/schema/ vs packages/db/schema/**
This is a layout deviation from the existing codebase pattern (`packages/db/schema/` is the drizzle-kit target per `drizzle.config.ts`: `schema: './schema'`). The new `packages/db/src/schema/` directory is NOT referenced by `drizzle.config.ts` and will NOT be picked up by `drizzle-kit generate` or `drizzle-kit push`. The existing `packages/db/schema/index.ts` still exports only `export {}` (empty), so nothing is broken today. However, the new table is invisible to drizzle-kit tooling until either: (a) `drizzle.config.ts` is updated to include `./src/schema`, or (b) the new schema is re-exported through `./schema/index.ts`. This is a latent tooling gap. Not a test-time blocker because the SQL migration is applied directly, but it is a debt item that must be resolved before drizzle-kit diff/push workflows are used. Logged as a carry-forward, not a hard blocker for this task.

**Concern 5 — Symlink packages/db/src/migrations -> ../migrations**
The symlink is an absolute path symlink (`/home/user/monopilot-kira/packages/db/migrations`), which means it will break on any developer machine or CI environment where the repo is checked out at a different path. Symlinks should use relative paths. Additionally, symlinks are not preserved by all Git clients (Windows Git without `core.symlinks=true` materialises them as text files containing the target path). This is a latency risk in CI and on Windows workstations. Acceptable only because the symlink's sole purpose is to satisfy `packageRoot` resolution in the test, and the test already skips when `DATABASE_URL` is absent (the integration path). The non-integration drizzle schema tests do not use the symlink. Risk is low-severity but should be cleaned up: the test's `packageRoot` should be computed relative to the actual migrations directory rather than relying on a symlink.

**Concern 6 — SQL correctness review**

- cohort CHECK: `cohort in ('canary', 'early', 'general')` — all three values present. PASS.

- R13 columns: NOT included. The PRD spec for T-038 does not list R13 columns for this table (§8 specifies exactly 5 columns; the implementer extended to 8 per the task JSON which adds cohort, status, failure_reason). R13 absence is intentional and correct per task scope.
- RLS: NOT applied to tenant_migrations. The SQL does not call `ALTER TABLE ... ENABLE ROW LEVEL SECURITY`. This is appropriate — this is an orchestration control-plane table, not a per-org data table. The `app.current_org_id()` RLS pattern from 002-rls-baseline applies only to `organizations` and `users`. No RLS needed here.
