Notes for implementer:
- Mutation interface exported from index.ts (see scope)
- Tests use fake-indexeddb (jsdom environment) for IndexedDB polyfill
- All 4 acceptance criteria have test coverage:
  - AC1: empty queue start + enqueue/listPending basic contract
  - AC2: idempotent dedup by transaction_id (re-enqueue same ID = no-op, keeps first entry)
  - AC3: remove by transaction_id + FIFO ordering (sorted by created_at ASC)
  - AC4: UUID v7 time-ordering property (lexicographic sort matches insertion order)
- generateTransactionId must use uuid v7 (time-ordered), NOT v4
- Key implementation details to watch:
  - Use idb-keyval with key pattern `mut:${transaction_id}` for mutations
  - listPending must return sorted by created_at (ascending)
  - enqueue is idempotent: same transaction_id twice = 1 entry (no overwrite of existing)
  - remove is idempotent: removing non-existent ID is no-op
  - Preserve full body (nested objects, arrays, null/undefined all OK)
- No network code (that is T-044)
- No localStorage fallback (IndexedDB mandatory per PRD §5)
