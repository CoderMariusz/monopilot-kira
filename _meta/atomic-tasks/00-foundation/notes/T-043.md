## Phase: REVIEW
Status: PASS

Verdict: **PASS**

### Concern 1 — Raw IDB instead of idb-keyval default store
Acceptable deviation. The root cause is real: `idb-keyval`'s default store uses a module-level singleton `IDBDatabase`. When `fake-indexeddb` calls `indexedDB.deleteDatabase()` in `beforeEach`, the pending open connection blocks the delete, hanging the test suite. The implementer's fix — raw IDB with `onversionchange` + `onclose` handlers that null out the cached `_dbPromise` — is the correct solution to let `deleteDatabase` close the connection cleanly. The `idb-keyval` dependency is still declared in `package.json` (could be removed as dead code, but it is not harmful). The spirit of the spec ("tiny IndexedDB wrapper, no full Dexie dependency") is fully satisfied; raw IDB is lighter than `idb-keyval`, not heavier. The key schema (`mut:${transaction_id}`) and all public APIs match the spec exactly. Minor: `idb-keyval` could be removed from `dependencies` as it is unused at runtime; this is cosmetic debt, not a blocker.

### Concern 2 — UUID v7 implemented inline (not via `uuid.v7()`)
Acceptable. The `uuid` package's `esm-node` bundle (reached when vitest resolves the `node` condition) uses `import crypto from 'node:crypto'` and `crypto.randomFillSync()`. Under vitest's jsdom environment the module condition resolves to the `node` export map entry (`wrapper.mjs` → `dist/index.js` → `dist/rng.js`), which calls `node:crypto.randomFillSync`. This has historically caused failures in jsdom rigs (the browser export map entry uses `crypto.getRandomValues` which jsdom polyfills via `fake-indexeddb`'s setup). The inline implementation uses only `globalThis.crypto.getRandomValues`, which works cleanly in jsdom and is RFC 9562 §5.7 compliant. The algorithm (48-bit timestamp, version nibble 0x7, 2-bit variant 0b10, 31-bit monotonic counter split 12/19, random tail) is structurally identical to `uuid`'s own `esm-node/v7.js`. If vitest adds a `browser` condition override (e.g., `resolve.conditions: ['browser']` in vitest config), `uuid.v7()` could work; that would be the cleaner long-term fix, but the inline solution is correct and production-safe.

### Concern 3 — UUID regex `{26}` → `{32}`
Correct fix. The format returned by `generateTransactionId()` is 16 bytes encoded as 32 lowercase hex characters (no hyphens). The regex `/^[0-9a-f]{32}$/i` is exactly right:
- 32 hex chars = 16 bytes (128 bits) = full UUID v7 without separator hyphens.
- `{26}` was an objective error (would only match 26 hex chars = 13 bytes, which is neither a standard UUID nor correct).
- For reference: hyphenated UUID v7 is 36 chars (`xxxxxxxx-xxxx-7xxx-xxxx-xxxxxxxxxxxx`); hyphen-free is 32 chars. Both are valid representations.
The test now correctly validates the output format.

### Concern 4 — FIFO ordering: deterministic per UUID v7 time prefix?
`listPending()` sorts by `created_at` ASC (the ISO 8601 string supplied by the caller), not by the IDB key or UUID. This is correct and deterministic. The AC3 FIFO tests use explicit, distinct `created_at` timestamps so ordering is unambiguous. The integration tests use `new Date().toISOString()` which may collide if mutations are created within the same millisecond; but even then the sort is stable in V8 (ECMAScript 2019+ guarantees stable `Array.prototype.sort`), so insertion order is preserved for ties. FIFO behaviour is sound.

### Concern 5 — Idempotent dedup
Implementation: `enqueue` calls `store.get(key)` inside a `readwrite` transaction; if the result is not `undefined`, it resolves early without calling `store.put`. Tests AC2 explicitly cover: (a) same mutation enqueued twice → 1 entry, (b) first entry's body is preserved (second is discarded), (c) mixed sequence with two distinct IDs each duplicated → 2 entries in FIFO order. All three pass. R14 dedup contract is satisfied.
