### Token-lookup strategy (BLOCKER decision for GREEN)
RED tests assume the middleware can verify a token in <10ms.  The decision between two strategies is left to GREEN:

1. **Index by `scim_token_last_four`**: extract last four chars of bearer plaintext, `SELECT scim_token_hash FROM tenant_idp_config WHERE scim_token_last_four = $1`. Returns at most a handful of rows (typically 1). Run argon2.verify against each. Avoids O(N) full-table scan; preserves <10ms budget for invalid tokens because last_four is a partial collision filter.
2. **Scan all hashes**: SELECT all `scim_token_hash` rows and verify against each. O(N) argon2 cost — viable only for small N. Will exceed 10ms once N > ~50 tenants.

**Recommendation: strategy (1).** RED test 6 (verify-path timing) will fail strategy (2) at scale. Strategy (1) requires that `scim_token_last_four` be indexed; GREEN must add `CREATE INDEX IF NOT EXISTS tenant_idp_config_scim_last_four_idx ON tenant_idp_config (scim_token_last_four)` (additive ALTER) — this is a small additive fix, can land in the same migration that adds `users.deleted_at`.

**Cross-tenant guard (red line):** if multiple tenants somehow share the same `scim_token_last_four` AND the same hash matches more than one row (collision or operator misconfiguration), middleware MUST reject with 401 — never pick one tenant arbitrarily. Test coverage for this disambiguation is not in this RED set; if GREEN wants belt-and-braces, add a 4th AC1 sub-test asserting "two tenants with identical last_four → 401 for both".

### Org-context wiring (T-062 carry-forward)
T-011 deferred `app.set_org_context` wiring out of middleware (edge-runtime cannot use `pg`). T-013 routes are Node-runtime route handlers (not middleware), so they CAN call `set_org_context` directly. After token verification:

1. Resolve `org_id` from the matched `tenant_idp_config.tenant_id` → `organizations.id` (one-to-one in current schema; if multiple orgs per tenant exist, the SCIM contract requires the token to identify a single org).
2. Generate a fresh `session_token` UUID (per request — NOT cached).
3. Owner-pool: `INSERT INTO app.session_org_contexts (session_token, org_id) VALUES ($1, $2)`.
4. App-pool: `BEGIN; SELECT app.set_org_context($1, $2); ... INSERT/SELECT/UPDATE ...; COMMIT`.
5. Owner-pool cleanup: `DELETE FROM app.session_org_contexts WHERE session_token = $1` (or rely on `app.active_org_contexts` ON DELETE CASCADE / TTL job).

This is the T-062 carry-forward pattern: SCIM is a privileged surface but uses RLS for cross-tenant isolation (defence in depth).
