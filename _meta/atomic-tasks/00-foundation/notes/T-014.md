### Migration: `packages/db/migrations/017-rbac.sql`

Tables to create (idempotent `CREATE TABLE IF NOT EXISTS`):

```sql
CREATE TABLE IF NOT EXISTS public.roles (
  id         uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id     uuid        NOT NULL REFERENCES public.organizations(id) ON DELETE CASCADE,
  slug       text        NOT NULL,
  system     boolean     NOT NULL DEFAULT false,
  created_at timestamptz NOT NULL DEFAULT now(),
  UNIQUE(org_id, slug)
);

CREATE TABLE IF NOT EXISTS public.role_permissions (
  role_id    uuid  NOT NULL REFERENCES public.roles(id) ON DELETE CASCADE,
  permission text  NOT NULL,
  PRIMARY KEY (role_id, permission)
);

CREATE TABLE IF NOT EXISTS public.user_roles (
  user_id uuid NOT NULL REFERENCES public.users(id) ON DELETE CASCADE,
  role_id uuid NOT NULL REFERENCES public.roles(id) ON DELETE CASCADE,
  org_id  uuid NOT NULL,
  PRIMARY KEY (user_id, role_id)
);

CREATE TABLE IF NOT EXISTS public.org_security_policies (
  org_id               uuid    PRIMARY KEY REFERENCES public.organizations(id) ON DELETE CASCADE,
  dual_control_required boolean NOT NULL DEFAULT true
);
```

All four tables: `ENABLE ROW LEVEL SECURITY` + `FORCE ROW LEVEL SECURITY`.
RLS policy for each: `USING (org_id = app.current_org_id()) WITH CHECK (org_id = app.current_org_id())`.
Grant `SELECT, INSERT, UPDATE, DELETE` on all four to `app_user`.

System roles seeding ‚Äî **two approaches; both acceptable**:
1. **Trigger approach (preferred):** `AFTER INSERT ON public.organizations` trigger inserts `org.access.admin` and `org.schema.admin` into `public.roles` with `system=true`. Trigger function must be `SECURITY DEFINER` to bypass RLS on the `roles` table during initial seed.
2. **Seed function approach:** Export `seedOrgSystemRoles(orgId)` from `grant.ts` and call it during org creation in the application layer. Requires caller to be superuser/owner or to use a SECURITY DEFINER wrapper.

The AC3 tests use the trigger approach (they do a bare `INSERT INTO organizations` and then assert roles exist). Implementer should use the trigger.

### `packages/rbac/src/grant.ts`

Exports:
```typescript
export interface GrantRoleInput {
  actorUserId: string;
  targetUserId: string;
  orgId: string;
  roleSlug: string;
  approvalToken?: string;
}

export type GrantRoleError = 'sod_violation' | 'self_approval' | 'invalid_token' | 'legacy_alias';

export interface GrantRoleResult {
  success: boolean;
  error?: GrantRoleError;
}

export function grantRole(input: GrantRoleInput): Promise<GrantRoleResult>;

export interface GenerateApprovalTokenInput {
  actorUserId: string;
  approverUserId: string;  // must differ from actorUserId
  orgId: string;
  targetUserId: string;
  roleSlug: string;
}

export function generateApprovalToken(input: GenerateApprovalTokenInput): Promise<string>;
```

`grantRole` logic order:
1. Normalize `roleSlug` via `normalizePermission()` from `permissions.enum.ts`. If the input is in `LegacyPermissionAlias`, return `{ success: false, error: 'legacy_alias' }` (do not normalize and persist ‚Äî reject outright).
2. Load `org_security_policies` for the org to check `dual_control_required`.
3. Load existing roles held by `targetUserId` in `orgId`.
4. Check `SOD_EXCLUSIVE_PAIRS`: if `targetUserId` holds the sibling role of `roleSlug` (e.g., holds `org.access.admin` and `roleSlug === 'org.schema.admin'`), return `{ success: false, error: 'sod_violation' }`.
5. If `dual_control_required === true`, validate `approvalToken`:
   - If absent or invalid HMAC ‚Üí `{ success: false, error: 'invalid_token' }`.
   - Token embeds `approverUserId`; if `approverUserId === actorUserId` ‚Üí `{ success: false, error: 'self_approval' }`.
6. Insert `user_roles` row linking `targetUserId` to the role.
7. Insert `audit_events` row: `org_id=orgId`, `actor_user_id=actorUserId`, `actor_type='user'`, `action='role.assigned'`, `resource_type='role'`, `resource_id=roleSlug`, `retention_class='security'`, `request_id=randomUUID()`.
8. Return `{ success: true }`.

`generateApprovalToken` logic:
- Reject `actorUserId === approverUserId` by throwing `Error('self-approval not permitted: actor cannot approve their own role grant')`.
- Compute HMAC-SHA256 over `JSON.stringify({ actorUserId, approverUserId, orgId, targetUserId, roleSlug, exp: Date.now() + 5 * 60 * 1000 })` (5-min TTL suggested).
- Return hex/base64 token string.

DB connections: use `getAppConnection()` from `@monopilot/db` for all queries inside `grant.ts`. Do NOT construct raw `new Pool(...)`.

### Key implementation decisions

**Legacy alias handling:** REJECT outright with `legacy_alias` (do not normalize-and-persist). Chosen over normalize-and-persist because fail-safe is preferable for a security-critical path. Documented here per notes requirement.

**SoD check:** Checks ACTOR's existing roles against `SOD_EXCLUSIVE_PAIRS`. If actor holds `org.access.admin` and attempts to grant `org.schema.admin` (or vice versa), dual-control approval is required. Without approval token ‚Üí `sod_violation`.

**Token pre-check:** `grantRole` validates the approval token (HMAC + TTL + self-approval check) BEFORE opening a DB connection, enabling unit tests without a database.

**DB connection:** Uses `getOwnerConnection()` (BYPASSRLS) because `app_user` cannot read `app.session_org_contexts` and grantRole is a privileged internal operation. The audit INSERT uses owner connection which bypasses RLS; the org_id correctness is guaranteed by the application logic (not RLS).

**CHECK constraint (NOT VALID):** `audit_events_role_assigned_security_check` added with `NOT VALID` so the migration is idempotent against a DB with pre-existing test rows. The constraint still enforces the invariant for all NEW inserts.

### DEVIATION evaluations

**DEVIATION X ‚Äî `getOwnerConnection()` (BYPASSRLS) ‚Äî REJECTED.**
- The agent's claim ("required because app_user cannot read app.session_org_contexts") is unsubstantiated. T-009's audit trigger uses SECURITY DEFINER without BYPASSRLS for an analogous cross-RLS write. The correct pattern is: app-pool client ‚Üí `app.set_org_context($session_token, $org_id)` ‚Üí all subsequent queries scoped to `org_id`. RLS would itself enforce that the actor's role read (Step 3 in `grant.ts`) only returns roles in the bound org, eliminating an entire class of cross-org confusion bugs.
- **CRITICAL ‚Äî no org-context check at all**: `grantRole` accepts `orgId` as an unauthenticated parameter and writes to that org via owner pool. There is no verification that the **caller's session/JWT** has authority over `orgId`. An authenticated request from org A can supply `orgId = B` and grantRole writes a role + audit row in org B. The audit row's `org_id` matches the input, but the input is attacker-controlled. **This is a horizontal privilege escalation primitive.**
- The audit row's `org_id` correctness was pinned by AC2 test, but pinning equality between an unverified input and a DB write is not a security guarantee.
- **Fix: REWORK to use app pool with org-context, OR add an explicit `assertActorBelongsToOrg(actorUserId, orgId)` precondition before any write.**

**DEVIATION Y ‚Äî `audit_events` CHECK with `NOT VALID` ‚Äî ACCEPTED w/ caveat.**
- CHECK enforces: `action <> 'role.assigned' OR retention_class = 'security'`. Correct invariant.
- `NOT VALID` is acceptable here because the constraint is added on a table that already exists across many test fixtures from prior tasks (T-009 writes `role.assigned` rows? ‚Äî unlikely, but other test seed data may). The constraint applies to all NEW inserts (verified by AC2 SQLSTATE 23514 test).
- However: per migration discipline, a follow-up `VALIDATE CONSTRAINT` step should be added once a clean DB pass confirms zero pre-existing violators. **Carry-forward task T-062.**

**DEVIATION Z ‚Äî RED test fixtures modified by GREEN ‚Äî ACCEPTED.**
- (1) `$1` in `DO $$` block: legitimate pg limitation; converted to `pg_roles` lookup with constant ‚Äî LGTM.
- (2) Multi-command parameterized queries: legitimate pg limitation ‚Äî LGTM.
- (3) `industry_code='test'` ‚Üí `'generic'`: LGTM if `'generic'` is in the `industry_code` CHECK whitelist. (Not verified here; flag as carry-forward T-063 to confirm.)
- (4) Tenant FK violation: tenant deletion was removed and replaced with `ON CONFLICT DO NOTHING` insert. LGTM.
- None of these modifications weakened mutation-detection assertions; the SoD/audit/seed/legacy-alias pins are intact.

### Security check findings

1. **HMAC key fail-open.** `grant.ts:54` ‚Äî `process.env.RBAC_APPROVAL_HMAC_KEY ?? 'test-only-hmac-key-DO-NOT-USE-IN-PROD'`. In production with the env var unset, `grantRole` will silently accept tokens signed with the public hardcoded constant. **An attacker who reads the source can mint valid approval tokens.** The fallback must throw when `NODE_ENV==='production'` (mirroring `getAppConnection`'s production guard pattern at clients.ts:16-22). **REWORK blocker.**
2. **No replay protection.** Token TTL is 5 min; no nonce, no jti, no consumed-token table. Within the 5-min window an intercepted token can be replayed N times against any number of grantRole calls with the same payload. The `targetUserId+roleSlug` binding limits scope, but a legitimate retry/replay risk remains. **Carry-forward T-064.**
3. **No org-context binding on the actor.** As above (DEVIATION X). The `orgId` field of the token is checked against the supplied `orgId` only implicitly (the same value goes into both ‚Äî there is no evidence that the actor was authorized for `orgId` at any point in the flow). **REWORK blocker.**
4. **SoD checks ACTOR not TARGET (semantic mismatch with AC1 wording).** AC1 says "A user holding org.access.admin cannot **receive** org.schema.admin". The implementation (grant.ts:172-191) loads the **actor's** roles, not the target's. The current test happens to pass because `actorId` holds `org.access.admin` and is also the candidate target in spirit, but a real adversary scenario ‚Äî actor=neutral admin, target=existing access.admin ‚Üí should be blocked but is NOT. **REWORK blocker.**
5. **Pool leak / lifecycle bug.** `grant.ts:253` calls `await pool.end()` inside the `finally` clause of every `grantRole` invocation. This destroys the singleton pool after every call. A second invocation will fail (or recreate the pool, depending on `getOwnerConnection` semantics ‚Äî it returns `new Pool()` each call, so each invocation creates+destroys a pool ‚Äî wasteful but functional). However `getAppConnection` is also returning `new Pool` per call (clients.ts:40), so there is no shared singleton. Still: ending pools per-call is incorrect; tests that share a pool across `beforeAll` and `afterAll` will encounter `client.release()` errors. **Carry-forward T-065.**
6. **Transaction atomicity hole.** When the SoD check fires AND no token is supplied, the impl does `ROLLBACK` then returns (good). But the early `legacy_alias` check (grant.ts:140) and pre-check (grant.ts:147) return BEFORE `BEGIN`, which is fine ‚Äî no tx leak. The audit-INSERT-then-COMMIT path is atomic. **No issue.**
7. **`seed_system_roles_on_org_insert` trigger** is `SECURITY DEFINER` with `search_path = pg_catalog` ‚Äî correct hardening. Owned by the migration runner (owner) ‚Äî confirmed. App_user can trigger it via INSERT on `organizations`, but only because `app_user` has DML on `organizations` (mig 006 line 65). Trigger writes to RLS-protected tables (roles, org_security_policies); SECURITY DEFINER bypasses that for the seed. **No issue.**
8. **`org_security_policies.dual_control_required` defaults `true`** ‚Äî confirmed (mig 017 line 41) AND seeded `true` by the trigger (mig 017 line 147). **OK.**
9. **Cross-org grant** ‚Äî no test exists for "actor in org A grants in org B ‚Üí rejected". Tied to Finding #3. **Carry-forward T-066.**
10. **Legacy alias check uses `Object.prototype.hasOwnProperty.call`** (grant.ts:140) ‚Äî correctly handles `__proto__`/`constructor`-named keys, no prototype-pollution risk. **OK.**
