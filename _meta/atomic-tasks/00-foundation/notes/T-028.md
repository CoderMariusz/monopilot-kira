### Submit-disabled mechanism ‚Äî test harness contract
The tests render `<ReasonInput>` and a sibling `<button data-testid="submit-btn">` inside a plain `<div>`:
```tsx
<div>
  <ReasonInput name="reason" minLength={10} placeholder="Why?" />
  <button data-testid="submit-btn" type="submit">Save change</button>
</div>
```
`ReasonInput` is expected to set `aria-disabled="true"` on that button while `length < minLength`, and
remove it (or set `"false"`) when `length >= minLength`. Two viable patterns:
1. **Context + `useEffect` portal**: ReasonInput exposes a React context; a separate `<SubmitGuard>`
   consumer wraps the button. Tests do NOT use SubmitGuard ‚Äî they use a raw `<button>` sibling ‚Äî so
   this approach requires ReasonInput to reach into the DOM via `useEffect` / `useRef` via a
   container query. Not recommended.
2. **Recommended ‚Äî compound `ReasonInput.Root`**: Expose `ReasonInput` as a compound root that
   renders a context provider; the parent `<div>` in the test is not ReasonInput-owned, so the button
   is always a sibling outside any context boundary. This also does not work without caller cooperation.
3. **Simplest ‚Äî `onValidChange` callback + caller wires aria-disabled**: The tests use ReasonInput
   without a wrapper, so this requires the test harness to add a wrapper. But tests are already written
   without such a wrapper.
4. **Actual simplest ‚Äî ReasonInput uses a `ref`-forward + bubbles validity state to DOM root via
   `data-*` attribute** + a `useEffect` that queries `[data-testid="submit-btn"]` in the nearest
   ancestor. Fragile.

**Resolution the test actually requires**: The most straightforward interpretation is that
`ReasonInput` re-renders an outer wrapper div and sets `aria-disabled` on **all** `button[type=submit]`
siblings via a React `useState` + `useEffect` walking `parentElement.querySelectorAll('[type=submit]')`.
Implementer must pick one approach and document it. The test does not prescribe the mechanism ‚Äî it only
asserts the observable DOM outcome.

### AC2 ‚Äî 9 chars ‚Üí aria-disabled="true"

> "Given minLength=10 and the user types 9 chars, when reading the parent form's submit button, then it is disabled with aria-disabled='true'"

Test refs: `AC2` describe block ‚Äî types `'123456789'` (9 chars exactly), then `toHaveAttribute('aria-disabled', 'true')`.

**Mutation proof (off-by-one `<` vs `<=`):** Impl uses `value.length >= minLength` for `isValid`, so `!isValid` means `length < minLength`. At 9 chars, `9 < 10` is true ‚Äî button disabled. If impl used `<=` (wrong), at 9 chars `9 <= 10` would be true, `isValid` false, still disabled ‚Äî **this particular mutation would NOT be caught by AC2 alone**. However, AC1 types exactly 10 chars and asserts enabled; with `<=` the button would still be disabled at 10, which AC1 **does** catch. Combined coverage is adequate. AC2 itself at 9 chars is unambiguous.

**Vacuous scan:** No vacuousness ‚Äî asserts specific attribute value.

### AC3 ‚Äî 11 chars ‚Üí enabled; backspace to 9 re-disables

> "Given the user types 11 chars, when reading the same button, then it is enabled"

Test refs: `AC3` describe block ‚Äî 3 tests: enabled at 11, counter shows `11/10+`, backspace from 11‚Üí9 re-disables.

**Mutation proof (parentElement walk fails silently):** If `useEffect` failed to find the parent/submit, the button would retain `aria-disabled="true"` from AC1's initial mount. At 11 chars the test asserts `aria-disabled` is `null` or `"false"`. A silent failure would leave it `"true"`, failing this assertion. **Caught.**

**Backspace / bidirectional mutation guard:** AC3 test 3 types 11 chars (confirm enabled), then types `{Backspace}{Backspace}` (back to 9) and asserts `aria-disabled="true"`. The `useEffect` depends on `isValid` and runs on every change ‚Äî it sets `aria-disabled="false"` (not removes it) when valid, then `"true"` when invalid. This bidirectional flip is explicitly verified. ‚úì

**Sibling-submit fragility note:** The `parentElement.querySelectorAll('[type=submit]')` walk is acknowledged as fragile in the notes. For this primitive's stated scope (consumed inside a caller-controlled wrapper) the test harness exactly matches the production usage pattern described in modals.jsx. Multiple-submit-sibling edge case is not tested, but is out of scope. Acceptable for a primitive.

**Vacuous scan:** Test 3 first confirms the enabled state before backspacing ‚Äî no vacuousness.
