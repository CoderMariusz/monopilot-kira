2. Wrong path (`../../db` → `../db/migrations`) — correct; `packages/server/../../db` resolves to `packages/db` (skipping `packages/`), so the real path is `../db/migrations` from the package root.

5. Missing db arg — correct; integration tests need schema-isolated dbClient to avoid writing to public schema.

### Concern 2: SELECT-first + INSERT ON CONFLICT DO NOTHING + re-SELECT — race safety
Pattern is correct for single-connection sequential calls. For true multi-connection concurrency, both callers can pass the initial SELECT (empty), both run the handler (double side-effect), INSERT is idempotent via DO NOTHING, and re-SELECT returns the committed winner's row. This is documented in the implementer notes as a known limitation requiring advisory locks or SERIALIZABLE for stricter guarantees. Acceptable for the task scope.

### Concern 3: UUID v7 validation
Implementation checks `id[14] === '7'` after full UUID-format regex. This correctly rejects UUID v4 (`randomUUID()`) and non-UUID strings. It does NOT check the RFC 4122 variant byte (high nibble of group 3 must be 8/9/a/b), so it would accept technically malformed UUIDs like `xxxxxxxx-xxxx-7xxx-0xxx-xxxxxxxxxxxx`. The JSON's AC3 only requires "version nibble != 7" to throw — the implementation satisfies this contract exactly. The test helper `isValidUUIDv7` is stricter (checks `[89ab]` variant) but is not used in any assertion path. Acceptable per AC wording.

### Concern 4: R14 contract — handler never re-runs on replay
The SELECT-first code path guarantees this: if a row exists with matching hash, `return row.response_json` executes before `handler()` is ever reached. R14 is satisfied on the replay path. PASS.

### Concern 5: Migration correctness
- Table `idempotency_keys` created with `CREATE TABLE IF NOT EXISTS` (idempotent). PASS.
- Columns: `transaction_id UUID PK`, `org_id UUID NOT NULL`, `request_hash TEXT NOT NULL`, `response_json JSONB NOT NULL`, `created_at TIMESTAMPTZ NOT NULL DEFAULT now()`. All required columns present. PASS.

### CRITICAL BUG: hashPayload key-sorting is broken for nested objects
`JSON.stringify(payload, Object.keys(payload).sort())` — when the second argument to `JSON.stringify` is an array, it acts as a **key whitelist**, not a sort instruction. Only top-level keys present in the array are included; all nested object keys are silently dropped. Result: `{ action: 'insert', data: { name: 'original' } }` serialises to `{"action":"insert","data":{}}` — identical to `{ action: 'insert', data: { name: 'modified' } }`. This causes false hash equality for any payload with nested objects, meaning `idempotency_conflict` will NOT be thrown when it should be, violating AC2.

Evidence: the AC2 test "should not mutate any row when idempotency_conflict is detected" uses `firstPayload = { action: 'insert', data: { name: 'original' } }` and `secondPayload = { action: 'insert', data: { name: 'modified' } }`. Both hash to the same value. The test passes vacuously (the `try/catch` swallows what is actually a CACHE HIT return, not a conflict throw), giving false green confidence. The other AC2 test ("should throw idempotency_conflict") uses flat payloads with different top-level values and correctly catches the conflict — masking the bug from overall test results.

Fix: replace with a recursive sort approach, e.g.:
```ts
function hashPayload(payload: Record<string, unknown>): string {
  const canonical = JSON.stringify(payload, (_key, value) =>
    value && typeof value === 'object' && !Array.isArray(value)
      ? Object.fromEntries(Object.entries(value).sort(([a], [b]) => a.localeCompare(b)))
      : value
  );
  return createHash('sha256').update(canonical).digest('hex');
}
```
This ensures deterministic key ordering at all nesting depths.
