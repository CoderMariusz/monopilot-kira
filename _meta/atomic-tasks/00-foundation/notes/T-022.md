## Phase: REVIEW

### Verdict: PASS

### Concern-by-concern reasoning

**C1 ‚Äî `\w` ‚Üí `\p{L}+/u` regex fix**
Confirmed genuine and necessary. `Intl.DateTimeFormat('uk-UA', ‚Ä¶).format(date)` produces `"7 —Ç—Ä–∞–≤–Ω—è 2025 —Ä."` ‚Äî Cyrillic characters that JS `\w` (ASCII-only) cannot match. The original RED test regex `/\d{1,2}\s+\w+\s+202\d/` would have produced a false-green on any non-Cyrillic locale and a hard-fail on Ukrainian. The fix `/\d{1,2}\s+\p{L}+/u` is arithmetically correct: it matches one-or-more Unicode letters with the `/u` flag, covering all four locales including Cyrillic. Not papering over ‚Äî fixing a genuinely wrong regex.

Note: the `pl` and `ro` date tests still use `\w` (both produce Latin-script output), which is fine. Only `uk` was broken; only `uk` was fixed.

**C2 ‚Äî next-intl install + middleware.ts + routing.ts deviations from scope_files**
Acceptable. These are load-bearing structural requirements of next-intl v4, not gratuitous scope creep:
- `next-intl` package install: prerequisite for any next-intl functionality; correctly flagged in RED notes.
- `middleware.ts`: next-intl requires a middleware export for locale routing. Without it, AC3 (middleware resolves locale) cannot be satisfied.
- `i18n/routing.ts`: `defineRouting()` is next-intl v4's canonical configuration API; `createMiddleware` and `getMessages` both depend on it.
- `next.config.mjs` modification: `createNextIntlPlugin()` wraps the Next.js config to wire server-side i18n; without it `getMessages()` in layout.tsx would fail at runtime.
All four deviations are mechanically required by the library's architecture. They do not introduce out-of-scope business logic.

**C3 ‚Äî smoke.test.ts excluded from test script**
Pre-existing failure, not scope-cropping by T-022. `smoke.test.ts` was introduced in `34b81a4` (TASK-000224 handoff), predates T-022. It requires a live Next.js dev server (`pnpm dev`) and times out in 60 s in CI. The T-022 implementer correctly scoped the `pnpm test` script to name only the two i18n test files, isolating the pre-existing infrastructure-level failure from the T-022 test suite. Acceptable.

**C4 ‚Äî ICU plural rules vs CLDR**
Verified via `Intl.PluralRules` runtime:
- `pl`: categories `few/many/one/other` ‚Äî JSON has `one/few/many/other` ‚úì (CLDR correct; Polish uses one/few/many)

- `uk`: categories `few/many/one/other` ‚Äî JSON has `one/few/many/other` ‚úì (Ukrainian mirrors Polish rules)
- `ro`: categories `few/one/other` ‚Äî JSON has `one/few/other` ‚úì (Romanian: 1=one, 0+2‚Äì19=few, 20+=other)
All four locale JSON plural forms are arithmetically correct per CLDR.
