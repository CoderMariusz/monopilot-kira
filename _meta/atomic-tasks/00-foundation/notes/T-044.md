Notes for implementer:
- File to create: `packages/sync-queue/src/flusher.ts`
- Exports required: `startFlusher()`, `stopFlusher()`, `flushOnce()`
- `startFlusher()` must be idempotent — use a module-level `isStarted` boolean flag; guard with `if (isStarted) return` on second call so only ONE `window.addEventListener('online', flushOnce)` call ever executes
- `stopFlusher()` must reset `isStarted = false`, call `window.removeEventListener('online', flushOnce)`, and clear the 30 s interval fallback timer
- `flushOnce()` must be sequential (NOT parallel) — use `for...of` over `await listPending()` result; if parallelised (e.g. `Promise.all`) the AC1 FIFO assertion on `mock.calls[0/1/2]` will catch it
- Per-request: `fetch(entry.endpoint, { method: entry.method, body: JSON.stringify(entry.body), headers: new Headers({ 'X-Transaction-Id': entry.transaction_id, 'Content-Type': 'application/json' }) })`
- 2xx OR 409 → `await remove(entry.transaction_id)` (treat dedup-success as success; AC2 verifies this)
- 5xx OR network error → do NOT remove; schedule backoff retry timer (first tier ≥ 300 000 ms); AC3 checks `vi.getTimerCount()` increased and that fetch is NOT called again within < 5 min window
- 4xx OTHER than 409 → also do NOT remove (human intervention required; red line)
- Backoff schedule: 5 min → 30 min → 2 h → capped at 12 h; mirrors packages/outbox worker convention
- 30 s interval fallback: `setInterval(flushOnce, 30_000)` started in `startFlusher()` — must be cleared in `stopFlusher()`
- Tests import from `../flusher.js` (ESM `.js` extension required for the package's `"type": "module"` setup)
- Tests use `vi.useFakeTimers()` in AC3 — ensure no `Date.now()` drift causes issues with the backoff schedule implementation
- Tests spy on `window.addEventListener` directly — a module-level stored function reference for the listener (not an inline arrow) is required so the same reference is used for both `addEventListener` and `removeEventListener`
