Notes for implementer:

### Migration
- File: `packages/db/migrations/019-pins.sql` (NOT `008-pins.sql`)
- Table DDL:
  ```sql
  CREATE TABLE IF NOT EXISTS public.user_pins (
    user_id        uuid        PRIMARY KEY REFERENCES public.users(id) ON DELETE CASCADE,
    pin_hash       text        NOT NULL,
    attempts_count int         NOT NULL DEFAULT 0,
    locked_until   timestamptz,
    last_attempt_at timestamptz
  );
  ALTER TABLE public.user_pins ENABLE ROW LEVEL SECURITY;
  ALTER TABLE public.user_pins FORCE ROW LEVEL SECURITY;
  -- RLS policy: USING (user_id IN (SELECT id FROM public.users WHERE org_id = app.current_org_id()))
  -- GRANT SELECT, INSERT, UPDATE, DELETE ON public.user_pins TO app_user;
  ```
  RLS choice: `user_id IN (SELECT id FROM users WHERE org_id = app.current_org_id())` matches the
  pattern from 018-password-history.sql and 011-departments.sql. The simpler `USING true` would
  not protect cross-org leakage if org context is ever mis-set; the org-scoped subquery is safer.

### Module: `packages/auth/src/verify-pin.ts`
Exports:
```ts
export async function setPin(userId: string, pin: string): Promise<void>
export async function verifyPin(userId: string, pin: string): Promise<true | false | 'locked'>
```

argon2 params:
```ts
import * as argon2 from 'argon2';
const ARGON2_OPTS = {
  type: argon2.argon2id,
  memoryCost: 65536,   // 64 MiB
  timeCost: 3,
  parallelism: 1,
};
```
These match the T-061 pattern. `argon2` is already in `packages/auth/package.json` (verified).

Lockout logic in `verifyPin`:
1. SELECT `pin_hash, attempts_count, locked_until, last_attempt_at` FROM `user_pins` WHERE `user_id=$1`
2. If `locked_until IS NOT NULL AND locked_until > now()`: return `'locked'`
3. Call `argon2.verify(pin_hash, pin)`
4. If verify succeeds: reset `attempts_count=0, locked_until=NULL`; return `true`
5. If verify fails: count failures in window = `attempts_count` (if `last_attempt_at > now()-10min`) else reset to 0; increment; `last_attempt_at = now()`
6. If new `attempts_count >= 6`: set `locked_until = now()+15min`; return `'locked'`
7. Else return `false`

(AC2 test spec: 5 failures → false, 6th → 'locked'. So condition to lock is when the attempt count
**after increment** reaches 6, i.e. `attempts_count >= 6` post-increment, or equivalently when the
failure being recorded is the 6th within the window.)

### DEVIATION evaluations

**α — RLS `USING (true)` on user_pins — P1 (carry-forward)**
- The migration explicitly weakens RLS: `using (true)` allows app_user to SELECT any user_pins row regardless of org context. Production paths (per T-011 middleware) DO set `app.current_org_id()`, so this is genuinely a test-only weakening that LEAKS to prod schema.
- Mitigated in practice because verify-pin.ts calls `getOwnerConnection()` (BYPASSRLS), so the policy is never actually exercised by production code. But: any future caller using `getAppConnection()` to query user_pins (e.g. an admin UI lookup) would silently leak cross-org PIN hashes.
- WITH CHECK guards writes only — reads are wide open.
- Proper fix: T-058 helper to set org context on test connection, then restore `using (user_id in (select id from public.users where org_id = app.current_org_id()))` per the RED-phase spec.
- **Verdict: ACCEPT with carry-forward task (T-062, P1).** Not a blocker because owner-conn path is the only consumer, but must be tightened before any future RLS-respecting consumer.

**β — setPin/verifyPin use getOwnerConnection() bypassing FORCE RLS — P1 (carry-forward)**
- Pattern matches T-014 grant.ts but rationale is weaker: setPin/verifyPin are user-self-service, not admin-privileged.
- **No org-scoping at the application layer** — `verifyPin(userId, pin)` accepts any UUID and queries it via BYPASSRLS pool. If the caller (server action) does not validate `userId === session.userId AND user.org_id === session.orgId`, this is a privilege-escalation surface.
- The function does not look up the user's org or compare to the session — it trusts the userId arg. That's reasonable for a library, but the CALLER contract must be documented.
- Additional bug: `pool.end()` in finally block is wasteful (pool factory creates fresh pool each call, so end() is no-op leak risk for connection-tracking). Not breaking but inefficient.
- **Verdict: ACCEPT with carry-forward (T-063, P1).** Add JSDoc precondition: "Caller MUST validate userId belongs to session org." Fix pool.end() leak.

**γ — Migrations 020 + 021 are scope creep — P0 (REWORK BLOCKER)**
- **020** adds nullable `slug` to `tenants`/`organizations` and `role` to `users`. PRODUCTION SCHEMA is `tenants(name)` only; existing tests (T-014 grant.test.ts, audit.integration.test.ts, rls.cross-org tests, departments.integration.test.ts) all seed WITHOUT slug. The verify-pin.test.ts seed is the lone outlier that invented `slug` columns.
- **021** sets `tenants.data_plane_url DEFAULT ''` and `organizations.industry_code DEFAULT 'generic'`:
  - `data_plane_url=''` is a **semantic lie**: empty string is not a URL, breaks any code that does `new URL(data_plane_url)`. The NOT NULL was protecting prod from orphaned tenants.
  - `industry_code='generic'` bakes a business decision (default industry) into the schema layer; PRD §8 places industry routing at the application layer.
- Industry-standard pattern (used by T-014 GREEN and 5+ other tests): test seeds insert `industry_code='generic'` explicitly. verify-pin.test.ts ignored this convention.
- 020 and 021 violate `out_of_scope: "Do not modify files outside the scope_files list"` — neither file was in the T-016 scope_files (only 019-pins.sql, verify-pin.ts, verify-pin.test.ts).
- **Verdict: REWORK — REVERT 020 + 021.** Fix verify-pin.test.ts seed (5–10 line change) to match the existing pattern: drop `slug` from tenants/organizations INSERTs, add `data_plane_url='https://test.invalid'` to tenants, add `industry_code='generic'` to organizations, drop `role` from users (or keep — users.role exists in 017-rbac).

**δ — last_attempt_at window logic + race condition — P2**
- The impl uses `BEGIN ... SELECT ... FOR UPDATE ... UPDATE ... COMMIT` (verify-pin.ts:80–166). `FOR UPDATE` row lock prevents concurrent verifyPin races on the same user_id — second tx blocks until first commits. Default READ COMMITTED is sufficient because of the row lock.
- Window-boundary race: only one tx at a time can touch the row, so the "reset attempts_count if last_attempt_at older than 10min" branch is atomic with the increment.
- **Verdict: ACCEPT.** Race-safe by design.
