## Phase: REVIEW
Status: REWORK
Reviewer agent type/model: claude-sonnet-4-6 (2026-05-07)

Findings:

### Concern 1 — Three RED test fixes

**Bug 1: `app.` → `${schemaName}.app.` three-part identifier substitution**
Confirmed real and correctly fixed. PostgreSQL does not support three-part identifiers (database.schema.object). The test's `public.` substitution was correct; substituting `app.` would have produced invalid references like `ci_audit_xxx.app.current_org_id()`. The fix — leaving `app.` unchanged so it remains a shared global schema — is the right solution. Verified in test lines 80-85.

**Bug 2: 10-item values array for 9 `$N` params + wrong actor_type binding**
Confirmed real and correctly fixed. Original INSERT had 9 column slots (no `occurred_at`) but 10 values, AND the ordering put `'user'` before `'impersonation'`, so `$3` (actor_type) bound to `'user'` not `'impersonation'` — the test would have inserted with actor_type='user' (no trigger) and then been surprised the rejection didn't happen. Fix removes the spurious item and correctly maps `$3 = 'impersonation'`. Verified at test lines 389-394.

**Bug 3: RLS COUNT(*) = 2 vs other rows in same schema**
Fix is technically correct — filtering `WHERE org_id IN ($1, $2)` does return exactly 2 for the two org_ids created in that test. However, this fix changes the nature of what is being tested. The original intent was "RLS isolation" (app_user sees only own-org rows), but the fixed test is running as the superuser/pool connection owner, not as `app_user`. There is no `SET LOCAL ROLE app_user` or `set_org_context(...)` call before the COUNT. The test now just verifies that rows can be filtered by `org_id` — it does NOT test RLS enforcement. The fix resolves the crash but leaves AC2/AC3-adjacent RLS isolation unverified. This is a gap, not a block (RLS logic itself in the migration is correct), but the test name claims "enforces RLS isolation" while delivering only a row-count sanity check.

### Concern 2 — Append-only enforcement

Migration correctly issues:
```sql
grant select, insert on public.audit_events to app_user;
revoke update, delete on public.audit_events from app_user;
```
Pattern matches 002-rls-baseline (e.g., `revoke all on app.session_org_contexts from app_user`). Order is GRANT then REVOKE, which is safe.

**Critical gap in tests:** The UPDATE-rejection test (lines 312-335) does `expect(hasUpdateRestriction || true).toBe(true)` — always passes regardless. The DELETE-rejection test (lines 337-357) asserts `constraints.length > 0` — also always passes. Neither test actually attempts `SET LOCAL ROLE app_user; UPDATE ...` to confirm the REVOKE works at runtime. AC2 is satisfied by the migration logic, but the test suite does not exercise the REVOKE path live.

### Concern 3 — Impersonation guard trigger

Trigger is BEFORE INSERT, raises P0001 on `actor_type = 'impersonation' AND impersonator_id IS NULL`. Trigger body is correct.

**Missing: SECURITY DEFINER with restricted `search_path`.** The function `public.audit_events_impersonation_guard()` is declared as plain `language plpgsql` with no `security definer` and no `set search_path`. Compare with 002-rls-baseline: `app.set_org_context` and `app.current_org_id()` are both `security definer set search_path = pg_catalog`. The 005-tenant-idp-config seed trigger also uses `security definer set search_path = pg_catalog`. The impersonation guard trigger is a BEFORE INSERT running with definer-level permissions; without `security definer` + restricted `search_path`, a user with CREATE SCHEMA access could shadow `public` to subvert the guard. For a security-critical table this is a real vulnerability. **This is a REWORK item.**

4. ROLLBACKs to restore the superuser session.

### REQUIRED 3 — DELETE-rejection test (same fix as above)

Fully implemented. The new test:
1. Seeds `app.session_org_contexts` with tokens for org1 and org2 (referencing real organization rows).
2. Grants `app_user` USAGE on the test-isolated schema (needed because the migration GRANT applies to table privileges, but schema USAGE for the ci_audit_* schema must be granted explicitly in the test).
3. Opens two successive transactions: each does `SET LOCAL ROLE app_user` + `SELECT app.set_org_context(token, orgId)` + `SELECT COUNT(*) FROM audit_events WHERE resource_id IN (...)`.
