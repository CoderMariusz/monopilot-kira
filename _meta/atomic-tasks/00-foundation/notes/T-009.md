### Concern 1 — Three RED test fixes

**Bug 1: `app.` → `${schemaName}.app.` three-part identifier substitution**
Confirmed real and correctly fixed. PostgreSQL does not support three-part identifiers (database.schema.object). The test's `public.` substitution was correct; substituting `app.` would have produced invalid references like `ci_audit_xxx.app.current_org_id()`. The fix — leaving `app.` unchanged so it remains a shared global schema — is the right solution. Verified in test lines 80-85.

**Bug 2: 10-item values array for 9 `$N` params + wrong actor_type binding**
Confirmed real and correctly fixed. Original INSERT had 9 column slots (no `occurred_at`) but 10 values, AND the ordering put `'user'` before `'impersonation'`, so `$3` (actor_type) bound to `'user'` not `'impersonation'` — the test would have inserted with actor_type='user' (no trigger) and then been surprised the rejection didn't happen. Fix removes the spurious item and correctly maps `$3 = 'impersonation'`. Verified at test lines 389-394.

**Bug 3: RLS COUNT(*) = 2 vs other rows in same schema**
Fix is technically correct — filtering `WHERE org_id IN ($1, $2)` does return exactly 2 for the two org_ids created in that test. However, this fix changes the nature of what is being tested. The original intent was "RLS isolation" (app_user sees only own-org rows), but the fixed test is running as the superuser/pool connection owner, not as `app_user`. There is no `SET LOCAL ROLE app_user` or `set_org_context(...)` call before the COUNT. The test now just verifies that rows can be filtered by `org_id` — it does NOT test RLS enforcement. The fix resolves the crash but leaves AC2/AC3-adjacent RLS isolation unverified.
