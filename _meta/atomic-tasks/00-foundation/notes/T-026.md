  ```ts
    id: string;
    label: string;
    canEnter?: () => boolean;   // absent = implicitly enterable
  }

  interface StepperProps {
    wizardId: string;            // key for stepperStore
    steps: StepDef[];

    cancelSlot?: React.ReactNode; // render-prop pattern for Cancel button

  }
  ```

#### Issue 1 — canEnter tab-click interception: partial mutation gap

In `Stepper.tsx` the click guard for locked tabs calls `e.preventDefault()` but does NOT call `e.stopPropagation()`. Radix `Tabs.Trigger` with `activationMode="manual"` also guards via `onValueChange`, which is correct as defense-in-depth. However, `e.preventDefault()` alone on a `<button>` inside Radix does NOT reliably prevent Radix's internal synthetic event from firing in all versions. The test at line 540 (`expect(onChange).not.toHaveBeenCalled()`) currently passes because `onValueChange` also has the guard (line 67 of Stepper.tsx). If that second guard were accidentally removed, the click-only preventDefault would be unreliable. The test would NOT catch this regression because it only tests the combined behavior, not the individual guards.

This is not a test mutation failure — the test correctly tests the observable behavior — but it is a fragility: the onClick guard alone is insufficient on a Radix Trigger.

#### Issue 2 — AC4 density assertion is vacuous

Test at line 660-675 (`RTL fallback: Stepper uses tokens.css density vars`) checks `root` exists but never asserts `data-density` attribute is set. If the implementer removed `data-density="default"` from Stepper.tsx, the test would still pass. The intent was to verify density tokens; the assertion does not.

#### Issue 3 — ESC test: focus may not propagate to stepper-root

In the ESC test (lines 388-410), `(stepperRoot as HTMLElement).focus()` is called on a `div` with `tabIndex={-1}`. Then `user.keyboard('{Escape}')` fires. The keydown is attached to `stepper-root` via `onKeyDown`. This relies on the stepper-root being focused (or a child being focused) so the event fires on it. With `userEvent.keyboard`, the event is dispatched to the currently focused element. If `stepper-root` itself has focus, the onKeyDown fires on it directly. This is correct. However, the test does not verify focus landed on stepper-root before pressing Escape — if focus silently failed (JSDOM focus quirk), the keyboard event might dispatch elsewhere. The test still passes in practice because `userEvent.setup()` + `.focus()` + `.keyboard()` sequence works in JSDOM. Flag as low-risk but noted.
