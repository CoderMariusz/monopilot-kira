- Function signature:
  ```ts
  evaluateTransition(
    workflowDef: WorkflowRule,
    currentState: string,
    requestedTransition: string,
    context: Record<string, unknown>
  ): Promise<{
    allowed: boolean;
    nextState: string | null;
    guard_failed: string[] | null;
    actions: Action[];
  }>
  ```
- Workflow JSON shape:
  ```ts
  {
    rule_id: string;
    rule_type: 'workflow';
    states: string[];
    initial_state: string;
    transitions: Array<{
      from: string;
      to: string;
      trigger: string;
      guards: Array<{ field: string; op: string; value: unknown }>;
      actions: Array<{ type: 'emit_event' | 'set_field' | 'call_service'; payload: object }>;
    }>;
  }
  ```
- Guard evaluation: each guard checks `context[field] op value` (op: '=', '!=', '>', '<', etc.)
- Sentinel order:
  - No transition matches `(currentState, requestedTransition)` → return `guard_failed: ['no_matching_transition']` (AC2 exact)

  ```ts
  switch (rule.rule_type) {

      // Extract { currentState, requestedTransition, context } from event
      // (or from rule.definition + event payload — implementer's choice)
      return evaluateTransition(rule, currentState, requestedTransition, context);
    ...
  }
  ```

Open questions / blockers (for implementer):
- **Rule.executor schema mismatch**: T-018's `Rule` interface has `triggers/conditions/actions`; the new state-machine workflow needs `states/initial_state/transitions`. AC4 tests pass a hybrid Rule shape (both classic and state-machine fields). GREEN must extend the `Rule` discriminated union OR introduce a `WorkflowRule` extending `Rule`. The cast `as Parameters<typeof executeRule>[0]` in AC4 tests reflects this — implementer should formalize the type.
- **Outbox event types for workflow actions**: emit_event payloads in the fixture use `wo.ready` and `quality.recorded` — both already in the 15-event CHECK (003 + 023). If a workflow needs to emit a workflow-specific event (e.g. `workflow.transitioned`), a new migration extending the CHECK is required (out of scope for T-035 per "Do not invent new rule_type values" red line; event types are separate from rule_types but still constrained).
- **Where the executor reads (currentState, requestedTransition, context) from**: choose between (a) event payload fields, (b) dedicated executor entrypoint, or (c) rule.definition_json. Tests pin them in `event.currentState`/`event.requestedTransition`/`event.context`, but GREEN may refactor — only the spy + no-throw assertions are load-bearing on AC4.

## Phase: GREEN (implementer)
Status: done

Files added/changed:
- `packages/rule-engine/src/workflow.ts` [created] — `evaluateTransition` + `WorkflowRule`/`WorkflowTransition`/`WorkflowGuard`/`WorkflowAction`/`EvaluateResult` types. Pure JSON-driven evaluator: matches transition by `(from, trigger)`, evaluates declarative guards via `evalOp` (`=`, `!=`, `>`, `<`, `>=`, `<=`), returns sentinel `['no_matching_transition']` for missing path. Dry-run (resolved from `opts.dryRun ?? context.dry_run`) skips outbox INSERTs entirely; normal-mode INSERTs are gated on an explicit `opts.pool` (test contract: no pool → no writes, keeping unit tests pure).
- `packages/rule-engine/src/executor.ts` [modified] — workflow case now dispatches through `workflowModule.evaluateTransition` (namespace import keeps `vi.spyOn` live-binding compatible) when `rule.transitions` is populated. Reads `(currentState, requestedTransition, context)` from `event.currentState | event.current_state`, `event.requestedTransition | event.requested_transition | event.event_type`, and `event.context`. Hybrid Rule shape supports state-machine fields (`states`, `initial_state`, `transitions`) optionally; legacy workflow rules (no transitions) keep the original cascading-style firing → preserves all 14 T-018 tests. Added `ExecuteRuleOptions { pool? }` for downstream callers to thread a real pg.Pool.
- `packages/rule-engine/package.json` [modified] — added `@monopilot/db` (workspace), `pg` 8.16.0, `@types/pg` ^8.20.0.

Implementation summary:
- evaluateTransition contract: returns `{ allowed, nextState, guard_failed, actions }`. `guard_failed=['no_matching_transition']` for missing path; failing-guard `field` names for guard-failures; `null` on success.
- Executor dispatch is fire-and-forget (`void evaluateTransition(...)`) so `executeRule` stays sync (existing 14 tests rely on sync result). The vi.spyOn assertion only requires the call to happen, not the awaited result.
- Side-effect path: outbox INSERTs only fire when (a) `dryRun=false` AND (b) explicit `opts.pool` provided AND (c) action.type='emit_event' with full `(org_id, event_type, aggregate_type, aggregate_id)` payload. AC3's dry-run delta=0 holds by construction.
