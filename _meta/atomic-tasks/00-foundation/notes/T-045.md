## Phase: REVIEW
Status: REWORK

Verdict: **REWORK** — three blocking issues prevent the AC contract from being satisfied.

### Concern-by-concern findings

**1. Three roles created (monopilot_app, monopilot_app_local, app_user)**
OK. The JSON spec explicitly calls for monopilot_app NOLOGIN template + monopilot_app_local LOGIN
per-env role. app_user is carried forward from T-007 as the test login role and inherits from
monopilot_app. Three roles is the correct shape. NOINHERIT on monopilot_app is safe: in PG16 the
GRANT's inherit option defaults to the MEMBER's inherit attribute; app_user has INHERIT=true, so
it correctly inherits monopilot_app's grants.

**2. getAppConnection() URL rewrite when DATABASE_URL_APP absent**
CONCERN. When DATABASE_URL_APP is not set, `clients.ts` rewrites the username to `app_user` and
hardcodes the password to `'app_user_test_password'` (via `APP_USER_PASSWORD ?? 'app_user_test_password'`).
This is acceptable only in a test-only environment where DATABASE_URL is known to be a local
non-superuser-routed URL. If DATABASE_URL points to a production or staging connection string with
superuser credentials, the rewrite still opens the connection but relies on a hardcoded test
password being the actual password of app_user in that environment. This is fragile and creates a
latent security hole: if APP_USER_PASSWORD is not set and the production app_user password is
NOT 'app_user_test_password', the app silently fails to connect (benign) but the intent is
misleading. **Required fix**: either throw when DATABASE_URL_APP is absent in non-test environments,
or document via an env-guard that this fallback is test-only. Acceptable for CI; not acceptable
as production behaviour.

**3. getOwnerConnection not re-exported from index.ts**
PASS. Confirmed: `src/index.ts` exports only `getAppConnection`. `getOwnerConnection` is absent.
Comment in index.ts explains the intentional omission.

**4. .eslintrc.cjs no-restricted-imports — rule cannot fire (BLOCKING)**
FAIL. The file exists and is syntactically correct, but:
- `packages/db/package.json` has no `eslint` dependency and no `lint` script.
- The root `pnpm lint` only runs `pnpm --filter web lint` (apps/web ESLint v9 flat config).
- ESLint v9 does not load `.eslintrc.cjs` without `ESLINT_USE_FLAT_CONFIG=false`.
- There is no invocation path by which `pnpm lint` would ever evaluate this rule against
  packages/db source files.
- Tested: running `pnpm lint` does not touch packages/db at all.
- AC3 states "when 'pnpm lint' runs, ESLint reports no-restricted-imports and exits non-zero."
  This is **not achievable** in the current state.
**Required fix**: add `eslint` + a `lint` script to `packages/db/package.json` and hook it into
root `pnpm lint` (or the root `ci` script), OR migrate to an ESLint v9 flat config
(`eslint.config.cjs`) that apps/web's ESLint can pick up when scanning the monorepo.

**5. Idempotent role creation**
PASS. All `CREATE ROLE` statements use `DO $$ BEGIN ... EXCEPTION WHEN duplicate_object THEN NULL END $$`.
The `GRANT monopilot_app TO app_user` uses `EXCEPTION WHEN others THEN NULL`. The `FORCE ROW LEVEL
SECURITY` and `GRANT` statements outside DO blocks are naturally idempotent in Postgres.

**6. FORCE ROW LEVEL SECURITY — scope creep?**
PASS / IN SCOPE. The JSON AC1 explicitly requires "FORCE ROW LEVEL SECURITY is applied to
tenants/organizations/users." Not scope creep; it is required. Note: `ENABLE ROW LEVEL SECURITY`
was already applied to organizations/users in 002-rls-baseline.sql; 0010_app_role.sql adds the
FORCE variant and extends both to tenants (which lacked RLS entirely). This is consistent with
the AC1 requirement. The monopilot_app_local role is also in-spec per the JSON contract
(implementation contract item 1 explicitly names it).

**7. Pre-existing failure in r13-business-tables.test.ts (GREEN note)**
GREEN's note is inaccurate. `r13-business-tables.test.ts` passes cleanly (33/33) when run in
isolation. The intermittent failure observed in the combined run is
`__tests__/departments.integration.test.ts` (3 tests, connection pool exhaustion when run in
parallel with all other integration tests). Both are unrelated to T-045.

**8. Test run — 9/9 pass**
PASS. `pnpm --filter @monopilot/db test src/__tests__/app-role.test.ts` → 9/9 pass. Confirmed.

**9. ESLint rule fires on violation — NOT VERIFIABLE (BLOCKING)**
Cannot be verified. No ESLint binary is available in packages/db and no lint script exists. A
temp violation file in `packages/db/src/__tests__/` cannot be linted with the existing toolchain.
This is the same issue as concern #4.

### Additional issues found

**A. Migration never applied to test DB (BLOCKING)**
`0010_app_role.sql` has NOT been run against the test database (`forza_mes` local Postgres).
Evidence: `monopilot_app` role does not exist; `tenants` has `relrowsecurity=false,
relforcerowsecurity=false`; app_user has no membership in monopilot_app. The tests pass only
because AC2/AC4 tests check `pg_roles` attributes of `app_user` (created by the test's own
`beforeAll` DO block), not the schema state produced by the migration. AC1's migration-outcome
guarantee is untestable and unverified.

The `scripts/migrate.ts` uses `drizzle-kit push` which applies only the TypeScript Drizzle schema,
not the raw SQL files in `migrations/`. There is no runner for the raw SQL migrations.

**B. Migration sort order broken**
`0010_app_role.sql` sorts lexicographically BEFORE `002-rls-baseline.sql` (because "001" < "002"
at position 2, and "0010_" has "1" at position 3 vs "002-" having "2"). If any runner applies
files alphabetically, 0010 will execute before 002, but 0010 GRANTs on tables created by
001-baseline and modifies RLS context set by 002. This is a latent ordering hazard. Naming
should follow the existing `NNN-` convention (e.g., `006_app_role.sql`) or the migration runner
must apply files in explicit ordered sequence.

**C. AC2 row-visibility test missing (BLOCKING for AC2)**
The JSON AC2 requires: "app role connects without setting app.current_org_id, SELECT * FROM tenants
→ 0 rows; owner role → all rows." This SELECT-count test does NOT exist in app-role.test.ts. The
test only verifies `rolsuper=false` and `rolbypassrls=false` from `pg_roles`. The core RLS
enforcement assertion is absent.

**D. AC4 "every test connects via getAppConnection()" not enforced**
The JSON AC4 requires "every test connects via getAppConnection()." All existing integration tests
(`__tests__/baseline.integration.test.ts`, `__tests__/departments.integration.test.ts`, etc.)
connect directly via `new pg.Pool({connectionString: process.env.DATABASE_URL})` as the superuser.
None use `getAppConnection()`. This is not enforced and not tested.

**E. Describe block still labelled "(RED)"**
`app-role.test.ts` line 24: `describe('T-045 app-role connection split (RED)', ...)`. The `(RED)`
label was not updated after implementation. Minor cosmetic issue.
