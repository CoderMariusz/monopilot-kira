**C5 — pg.Pool suppressions: sized correctly or over-suppressed?**
REWORK REQUIRED. The suppressions are NECESSARY (pre-T-058 pool usages are real), but they are OVER-BROAD in scope. In outbox, server, schema-runtime, and db, the suppression is:
```js
{ files: [...test globs...], rules: { 'no-restricted-syntax': 'off' } }
```
Setting `no-restricted-syntax: 'off'` disables the ENTIRE rule — which contains BOTH the `pg.Pool` gate (T-058) and the `Reference.*` gate (T-046). Live test confirmed: a `"Reference.Foo"` literal placed in `packages/outbox/src/__tests__/_review_fixture.test.ts` exits 0 (no error). Same gap confirmed in server and schema-runtime test globs, and in db `__tests__/**` and `**/*.test.ts` globs. This means AC2 ("Reference.* literal violation fails lint") is NOT guaranteed in test files of those 4 packages.

Fix required: split `no-restricted-syntax` into two separate rule entries (or use `no-restricted-syntax: [error, only-the-pool-entry]`) so the pg.Pool suppression does not also silence the Reference.* gate. Alternatively, restructure so only the Pool selector is suppressed in test overrides.

**C6 — `no-unused-vars` off for TS files**
ACCEPTABLE AS DOCUMENTED. The base disables `no-unused-vars` for `*.{ts,tsx,mts,cts}` files, deferring to TypeScript's own `tsc --noUnusedLocals`. This is the standard pattern for TypeScript projects: the ESLint JS rule produces false positives on type exports, interface members, and enum values. The trade-off is explicit and documented in `base.mjs` comments. Teams that want stricter coverage can add `@typescript-eslint/no-unused-vars` in their local config. Risk is low because `tsc` catches the same violations at build time.
