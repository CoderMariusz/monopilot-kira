## HOTFIX — 2026-05-07 (Wave A consistency audit T-056 / P0)

**Gap identified:** The 2026-05-07 Foundation Wave A Consistency Audit (finding D, severity P2 → escalated P0) found that `Reference.Departments` was the only Wave A business table without Row Level Security. All other org-scoped tables (`public.organizations`, `public.users`, `outbox_events`, `audit_events`, `Reference.FieldTypes`, `Reference.DeptColumns`, `Reference.Formulas`, `Reference.Rules`, and the R13 placeholder tables) had ENABLE + FORCE RLS + `org_id = app.current_org_id()` policies. `Reference.Departments` was absent from the T-019 spec's RLS requirements and the omission was carried through GREEN, REVIEW, and CLOSEOUT phases.

**Root cause:** T-019 task spec did not list RLS as a scope item. The T-007 RLS baseline review noted only `public.organizations` and `public.users`; `Reference.*` tables were added to RLS coverage incrementally via T-010/T-016/T-018 but T-019 slipped through.

**Fix applied:**

`packages/db/migrations/011-departments.sql` — added idempotent RLS block (section 3):
```sql
ALTER TABLE "Reference"."Departments" ENABLE ROW LEVEL SECURITY;
ALTER TABLE "Reference"."Departments" FORCE ROW LEVEL SECURITY;
DROP POLICY IF EXISTS "Departments_org_isolation" ON "Reference"."Departments";
CREATE POLICY "Departments_org_isolation" ON "Reference"."Departments"
  FOR ALL TO PUBLIC
  USING (org_id = app.current_org_id())
  WITH CHECK (org_id = app.current_org_id());
GRANT SELECT, INSERT, UPDATE, DELETE ON "Reference"."Departments" TO app_user;
```

`packages/db/__tests__/departments.integration.test.ts` — added new `describe` block "011 departments RLS — cross-org isolation" with 2 tests:
- `RLS isolation: app_user with Org B context sees 0 rows from Org A departments` — inserts a dept in Org A as owner, switches to `app_user` with Org B context, asserts SELECT returns 0 rows.
- `RLS isolation: app_user with Org A context can see Org A department` — verifies the positive case (correct org context returns the expected row).
