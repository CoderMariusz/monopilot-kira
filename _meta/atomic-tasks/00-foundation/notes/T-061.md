- **HIBP mock setup blocker:** Tests stub fetch via `vi.stubGlobal('fetch', vi.fn())` with canned k-anonymity responses. The implementation must:
  - SHA1 hash the password
  - Send first 5 hex chars to `https://api.pwnedpasswords.com/range/{prefix}`
  - Parse response for remaining 35 chars (line-by-line, case-insensitive)
  - Honor `process.env.HIBP_DISABLED=true` for offline CI runs

Findings — per concern:

**C1. NIST SP 800-63B compliance**
- Min-12: PASS. `newPassword.length < 12` correctly enforces minimum.
- No upper limit: PASS. No max-length check exists; passwords of any length are accepted.
- No whitespace-only: PARTIAL FAIL. Passwords of 12+ whitespace chars (e.g., `"            "`) pass all checks — not in COMMON_PASSWORDS, HIBP fails-open. NIST SP 800-63B §5.1.1.2 SHOULD check for composition of entirely whitespace. This is a SHOULD, not a SHALL, so it is not a blocker for a P1 stub, but should be noted.
- No all-same-char: PASS (via COMMON_PASSWORDS). All 26 lowercase 12-char repetitions (`aaaaaaaaaaaa`–`zzzzzzzzzzzz`) are in the list. Numeric repetitions (`000000000000`–`999999999999`) also covered.

**C2. Common-password list (~200 entries)**
The list is an inline curated set, NOT the NIST/HIBP top-25K subset. Acceptable for a stub: the task spec explicitly states "small offline banned-words list" and says "HIBP check catches the rest." The list covers all NIST known-bad test vectors required by AC7. Not a blocker; document as deviation.

**C3. HIBP k-anonymity**
- PASS. SHA1 → `digest('hex')` produces lowercase hex; URL uses lowercase prefix (correct per HIBP docs).
- Suffix comparison: `lineSuffix.toUpperCase() === suffix.toUpperCase()` — redundant double-toUpperCase on `suffix` (already uppercase) but harmless.
- 2s AbortController timeout: PASS.
- Fail-open: PASS.
- Injectable via `hibp?: HibpLookup`: PASS. Exported and documented.

**C4. History check — k=5 argon2.verify loop / DoS surface**
- Loop is correct: `LIMIT 5` in SQL, then iterates `rows` (at most 5) with `argon2.verify()`.
- DoS risk is real but acceptable for a stub: 5 × argon2 verify ≈ 5 × ~100ms CPU = ~500ms max. Not a regression; acceptable documented trade-off.
- CRITICAL WEAKNESS: The AC3 reuse test uses a stub hash (`$argon2id$v=19$m=65540,t=2,p=1$stub$hash`) that causes `argon2.verify()` to throw "Output is too short". The catch block treats this as no-match, so `checkPasswordHistory` returns `false`. The test assertion `if (result.ok === false) { expect(result.reasons).toContain('reused') }` is VACUOUSLY SATISFIED — the test never actually proves that reuse is detected. This is a test-quality bug: AC3 is not meaningfully verified.

**C5. `recordPasswordHistory` DELETE trim query**
PASS. Query is `DELETE FROM password_history WHERE user_id = $1 AND id NOT IN (SELECT id FROM password_history WHERE user_id = $1 ORDER BY created_at DESC LIMIT 5)`. Correct structure: keeps the 5 most recent, deletes the rest. Separate `id` PK (vs. composite PK) enables this pattern correctly.

**C6. Migration 018 — RLS**

FK to `users(id) ON DELETE CASCADE` correct. Index `(user_id, created_at DESC)` correct. Migration is idempotent (`CREATE TABLE IF NOT EXISTS`, `DROP POLICY IF EXISTS`).

**C7. Fail-open on DB error in history check**
Intentional and documented. If `password_history` table does not exist (pre-migration deploy), sign-up is not blocked. Trade-off: a DB outage during the history check allows password reuse silently. For a stub consumed by T-011, this is acceptable. Document as known deviation.
