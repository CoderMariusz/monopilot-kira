### `packages/ui/src/Field.tsx` — implementation contract
Props: `{ name: string; label: string; hint?: string; required?: boolean; type: string; schema?: z.ZodTypeAny }`

- Internally calls `useFormContext()` (caller wraps in `<FormProvider>`).
- Uses RHF `<Controller>` to register the field.
- Mode must be `'onBlur'` — the FormWrapper helper in tests calls `useForm({ mode: 'onBlur' })`.
- On validation error: set `aria-invalid="true"` on the input element; render the resolver error message below the input (replacing hint text).
- When no error and hint is provided: render hint below the input.
- The inner input must NOT be a raw `<input>` — it must be wrapped by an `Input` component that carries `data-slot="input"` on its root element.
- Required asterisk: `<span aria-label="required" tabIndex={-1}>*</span>` rendered inside or adjacent to the label. Non-interactive (not button/a, no positive tabIndex).
- `aria-describedby` on the input must point to the error element's `id` when an error is present.
- Label must be associated to the input (`htmlFor` / `id` pair) so `getByLabelText` resolves.

### `packages/ui/src/Input.tsx` — hand-roll a minimal shadcn-style wrapper (RECOMMENDED)
Shadcn is not installed. Create a minimal wrapper:
```tsx
// packages/ui/src/Input.tsx
import React from 'react';

export interface InputProps extends React.InputHTMLAttributes<HTMLInputElement> {}

const Input = React.forwardRef<HTMLInputElement, InputProps>((props, ref) => (
  <span data-slot="input" style={{ display: 'contents' }}>
    <input ref={ref} {...props} />
  </span>
));
Input.displayName = 'Input';
export default Input;
```
This satisfies the AC1 assertion that walks ancestors looking for `data-slot="input"`.

### AC1 — Structural / visual / interactional parity

**AC verbatim:** "Given the Field primitive renders, when compared to the invite-modal field block in prototypes/design/Monopilot Design System/settings/access-screens.jsx:139-145, then it matches structurally (label on top, input below, optional hint/error below), visually (uses shadcn Input as the inner primitive — no raw `<input>`), and interactionally (validation runs on blur and on submit, error message replaces hint when present, `aria-invalid='true'` is set on error) — verified by Field.test.tsx"

**Test reference:** `Field.test.tsx` → `describe('AC1: structural, visual, and interactional parity...')` — 7 tests (note: notes.md says 6 but file has 7 including submit trigger).

**Mutation proofs:**

- *`aria-invalid` hardcoded `true`*: Caught. Test `'clears aria-invalid and error text when value becomes valid after blur'` (AC2 block) asserts `expect(input).not.toHaveAttribute('aria-invalid', 'true')` after round-trip to valid. Additionally the AC1 test `'sets aria-invalid="true" on the input...'` uses `toHaveAttribute('aria-invalid', 'true')` — exact string match, not boolean. A hardcoded `true` (boolean coerced to `"true"` by DOM) would accidentally pass this one, but the round-trip clear test in AC2 would catch it.
- *Error message NOT rendered on blur*: Caught. `'error message replaces hint text when validation error is present'` uses `waitFor` with `getByText('Must be a valid email')` — will timeout/fail if error never appears.
- *Hint NOT hidden when error appears*: Caught. Same test asserts `expect(screen.queryByText('Enter a valid work email')).not.toBeInTheDocument()` immediately after error appears.
- *`aria-invalid` string vs boolean*: `toHaveAttribute('aria-invalid', 'true')` asserts the string `"true"`. Field.tsx sets `aria-invalid={hasError ? 'true' : undefined}` — correct string literal.
- *Submit trigger missing*: Caught. Dedicated test `'triggers validation on submit as well as blur'` fires `fireEvent.click` on submit button without any prior blur and awaits the error text. Covers both blur AND submit triggers — no gap.

**Vacuous scan:** No `toBeDefined()` on DOM nodes, no `expect(true).toBe(true)`, no bare `toThrow()`. `expect(foundSlot).toBe(true)` is backed by a real DOM walk — not vacuous.

**shadcn Input wrapper:** Input.tsx is `React.forwardRef` rendering `<span data-slot="input" style={{display:'contents'}}><input .../></span>`. Field.tsx uses `<Input>` not `<input>`. Test walks `parentElement` chain asserting `data-slot="input"` — confirmed sound.

### AC2 — Zod schema rejection on blur → `aria-invalid` + error text below input

**AC verbatim:** "Given a Zod schema rejects the value, when the user blurs the field, then `aria-invalid='true'` is set and the resolver error message renders below the input"

**Test reference:** `Field.test.tsx` → `describe('AC2: Zod schema rejects value → blur → aria-invalid + error text below input')` — 4 tests.

**Mutation proofs:**

- *`aria-invalid` hardcoded `true`*: `'clears aria-invalid and error text when value becomes valid after blur'` asserts `.not.toHaveAttribute('aria-invalid', 'true')` after valid re-blur. Hardcoded `true` breaks this.
- *Error message position not verified as below*: `'renders error message below the input after blur with invalid value'` uses `compareDocumentPosition` to assert error follows input in DOM — catches any implementation that renders error above input.
- *`aria-describedby` not wired*: `'wires aria-describedby from input to the error element'` reads `input.getAttribute('aria-describedby')`, then asserts `container.querySelector('#' + id)` is non-null with matching textContent. Catches both missing attribute and wrong id.

**FormProvider setup:** `FormWrapper` correctly calls `useForm({ mode: 'onBlur', resolver: zodResolver(schema) })` and wraps children in `<FormProvider {...methods}>`. If Field forgot `useFormContext()` it would throw at render, causing test to fail immediately — not a silent pass.

**Vacuous scan:** All assertions are specific: `toHaveAttribute` with exact value, `not.toBeNull()`, `textContent.toContain(...)`. None are vacuous.

### AC3 — `required=true` → visible asterisk with `aria-label="required"`

**AC verbatim:** "Given `required=true`, when the field renders, then a visible asterisk is present with `aria-label='required'`"

**Test reference:** `Field.test.tsx` → `describe('AC3: required=true renders a visible asterisk...')` — 5 tests.

**Mutation proofs:**

- *Asterisk renders when `required=false`*: `'does NOT render asterisk when required is omitted or false'` queries `container.querySelector('[aria-label="required"]')` and asserts `toBeNull()`. Catches an impl bug where asterisk always renders.
- *`aria-label` value is wrong string*: `'asterisk has aria-label="required" on the wrapping span'` uses `container.querySelector('[aria-label="required"]')` — attribute selector must match the literal string `"required"`.
- *Asterisk is interactive*: `'asterisk is not interactive'` asserts tag is not `button`/`a`, and `tabindex === null || Number(tabindex) < 0`. Field.tsx sets `tabIndex={-1}` so this passes correctly.
- *`aria-label` literal value confirmed*: `screen.getByLabelText('required')` is the exact string — not a regex. Fails if impl uses `aria-label="Required"` (capital R).
- *Label association*: `'label associates with the input'` uses `getByLabelText(/Email address/)` which must resolve to `<input>` element. Field.tsx wires `htmlFor={inputId}` on label and `id={inputId}` on Input — confirmed.

**Vacuous scan:** `expect(tabIndex === null || Number(tabIndex) < 0).toBe(true)` — this is the one test that ends up asserting a computed boolean `true`. However the computation is real (attribute value check + Number coercion), and `tabIndex={-1}` in impl means the assertion is meaningful. It would catch `tabIndex={0}` or `tabIndex={1}`. Marginally more direct would be `expect(Number(tabIndex)).toBeLessThan(0)`, but the current form is not vacuous.
