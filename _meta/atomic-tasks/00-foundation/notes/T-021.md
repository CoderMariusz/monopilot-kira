Implementation summary:
- Code shape: `WIP-${process_suffix}-${seq7}` where `seq7 = ((Date.now()*1000 + counter) % 10_000_000).padStart(7,'0')`. Matches `/^WIP-MX-\d{7}$/` (AC1) and `/^WIP-SY-\d{7}$/` (AC3). Inverted form `<suffix>-WIP-<seq>` would fail the prefix-fixed regex → mutation caught.
- Aggregate identity: outbox row written with `aggregate_type='fg'`, `aggregate_id=fgId`, `event_type='fg.intermediate_code_changed'` — bound to the same tx as the UPDATE → mutation that emits before the UPDATE / for the wrong fgId is caught by AC1's `aggregate_id===bakeryFgId`.
- Active-window enforcement (M3): `loadActiveCascadeRule` returns null on no row → handler ROLLBACKs and returns silently. Only the expired-version M3 fixture is seeded; no row brackets now() → no UPDATE → mutation that strips the window check would fire the cascade and fail the assertion.
- Cross-org isolation (M4): every UPDATE/SELECT/INSERT carries `org_id = $1` from the input arg. Even though the test passes `pool=ownerPool` (RLS-bypassed), the explicit scope filter prevents cross-org writes — mutation that drops `AND org_id=$3` from the UPDATE would touch otherOrgId's FG row and fail M4-(a). The second M4 test (app-pool RLS sanity) is independent of handler behavior — it exercises the test fixture's `fg_org_context` policy.
- Atomicity (AC2): the handler issues `BEGIN`, then `lookupProcessSuffix` throws inside the tx if no Reference.ManufacturingOperations row matches (org_id, operation_name) — caught in the outer try/catch, which `ROLLBACK`s and rethrows. No UPDATE or outbox INSERT executes → outbox delta=0, FG row's intermediate_code_p1 stays NULL.
- Dry-run (M1): after loading the rule + computing `newCode`, the handler ROLLBACKs and returns. No SELECT/UPDATE/INSERT side effects on public.fg or public.outbox_events.
- `public.fg` table: kept the inline-test-fixture pattern (Option 1 in RED notes). No new migration; no STATUS.md migration slot reservation needed.
