## HOTFIX — 2026-05-07: AC3 test was vacuous (metadata-only)

**Audit finding**: `2026-05-07-foundation-wave-a-test-quality-audit.md` (lines 464–537 entry) identified that the entire AC3 block — 10 tests labelled "AC3.1–AC3.10 RLS isolation" — only queried `pg_class.relrowsecurity = true` and `pg_policies.length > 0`. It never connected as `app_user`, never called `app.set_org_context`, and never performed a cross-org SELECT. This is the **same "claims security but tests metadata" trap** that triggered T-009's two rework rounds.

**Why it was vacuous**: `relrowsecurity = true` only tells you RLS is enabled on the table. A policy that contains `USING (true)` — i.e., grants all rows to all roles — would still pass both metadata assertions while providing zero isolation. The test would remain green even if the `USING (org_id = app.current_org_id())` predicate were removed from the migration.

**What the hotfix does** (`packages/db/__tests__/r13-business-tables.test.ts`):

1. `beforeAll` now creates/configures the `app_user` role with a known test password and grants SELECT/INSERT/UPDATE/DELETE on the 5 placeholder tables. A second `pg.Pool` (`appPool`) is opened with `app_user` credentials.

2. The 10 metadata-only tests are replaced with **6 genuine tests**:
   - `AC3.sanity` — a single consolidated metadata check (relrowsecurity + pg_policies) kept as a secondary guard.
   - `AC3.1–AC3.5` (one per table) — each test:
     - INSERTs 1 row for org A and 1 row for org B as superuser.
     - Seeds two tokens into `app.session_org_contexts`.
     - Opens a transaction as `app_user`, calls `app.set_org_context(tokenA, orgA)`, asserts `count(*) = 1` (only orgA row visible).
     - Rolls back, opens a fresh transaction with `tokenB/orgB`, asserts `count(*) = 1` (only orgB row visible).
     - Cleans up seeded rows.

3. The test **would FAIL** if the RLS policy were dropped from the migration because:
   - With `FORCE ROW LEVEL SECURITY` and no policy, `app_user` sees 0 rows — the assertion `count = 1` fails.
   - With a permissive policy (`USING (true)`), `app_user` sees 2 rows — the assertion `count = 1` also fails.
   - Only a correctly scoped `USING (org_id = app.current_org_id())` policy produces `count = 1` in each context.

**Test count change**: 33 tests → 29 tests (10 metadata-only → 1 sanity + 5 genuine cross-org; net -4 because the 6 replacements consolidate what were 10 separate tests).
