### Findings per concern

**Concern 1 — Decision Option A (no 017 migration)**
CONFIRMED. Lines 44–53 of `011-departments.sql` contain all three required clauses verbatim:
- `alter table "Reference"."Departments" enable row level security;` (line 44)
- `alter table "Reference"."Departments" force row level security;` (line 45)
- `drop policy if exists "Departments_org_isolation" on "Reference"."Departments";` (line 47, idempotent guard)
- `create policy "Departments_org_isolation" ... using (org_id = app.current_org_id()) with check (org_id = app.current_org_id());` (lines 48–51)
- `grant select, insert, update, delete on "Reference"."Departments" to app_user;` (line 53)
Option A is correct and well-justified. No 017 migration is needed.

**Concern 2 — 9 new tests genuine? AC4 SQLSTATE pinning non-vacuous?**
CONFIRMED. The test file contains exactly 9 tests: 3 in the static `describe` block (no DB) + 3 in the AC1 pg_class block + 3 in the AC2/AC3/AC4 block. AC4 (lines 304–334) uses a `try/catch` around the INSERT, captures `(insertErr as { code?: string }).code` into `caughtCode`, then asserts `expect(caughtCode, 'expected SQLSTATE 42501 (insufficient_privilege)').toBe('42501')`. This is not bare `.rejects.toThrow()` — the SQLSTATE code is explicitly extracted from the pg error object and pinned with a strict `.toBe()` equality assertion with a descriptive message. The assertion is non-vacuous: if the INSERT succeeded (no error), `caughtCode` would be `undefined`, causing the test to fail. Red-line satisfied.

**Concern 3 — GRANT USAGE ON SCHEMA "Reference" TO app_user in beforeAll**
CONFIRMED necessary. The GRANT block in 011 line 53 grants DML on the *table* to `app_user`, but 011 never grants `USAGE` on the `"Reference"` *schema* to `app_user`. Without `GRANT USAGE ON SCHEMA "Reference" TO app_user`, `app_user` cannot resolve the schema name even if the table policy allows access. The `beforeAll` at line 169 (`await adminPool.query('grant usage on schema "Reference" to app_user')`) plugs this gap. This is also present identically in the existing `departments.integration.test.ts` RLS suite (line 272), confirming it is a known necessary workaround, not an oversight.

**Concern 4 — Tests use getOwnerConnection / getAppConnection (no raw pg.Pool)**
CONFIRMED. Both integration describe blocks obtain pools exclusively via `getOwnerConnection()` (lines 68, 147) and `getAppConnection()` (lines 69, 148) imported from `../test-utils/test-pool.js`. No `new pg.Pool` anywhere in the file. Red-line satisfied.

**Concern 5 — AC3 asserts exactly 7 rows in Apex context**
CONFIRMED. The beforeAll seeds exactly 7 codes (`core, technical, packaging, mrp, planning, production, price`) via a typed `apexCodes` const array (lines 194–202). AC3 (lines 275–300) asserts `result.rows.toHaveLength(7)` AND uses `Set` equality against all 7 codes. Consistent with T-019 seed.

**Concern 6 — No regression in original 12-test suite**
CONFIRMED. `pnpm --filter @monopilot/db test` output shows `departments.integration.test.ts (12 tests | 7 skipped)` — still 12 total, unchanged. All ECONNREFUSED failures in that file are pre-existing environment failures (no Postgres in CI), not regressions from T-056.

**Concern 7 — pnpm --filter @monopilot/db test run result**

The 6 integration tests skip/fail only due to ECONNREFUSED (no Postgres in this environment) — identical to all other integration test suites. This is pre-existing, not introduced by T-056.
