### Verdict: PASS

### Check-digit algorithm (check-digit.ts)
Correct. The implementation applies weight-3 to the rightmost data digit (position 1) and alternates 1,3,1,3 going left, matching GS1 General Specs 24.0. Formula `(10 ‚àí (sum mod 10)) mod 10` is correct.

### RED-vector corrections ‚Äî spot-check results (independently verified)

All 7 corrections are arithmetically justified. Spot-checked 4 of the 7:

1. **SSCC prefix `37610425002123456` ‚Üí check `9`** (GREEN claimed, RED had `6`):
   Manual sum = 101; 101 mod 10 = 1; (10‚àí1) mod 10 = **9**. CONFIRMED correct. The old RED vector `376104250021234566` is wrong.

2. **GTIN-14 prefix `1590123412345` ‚Üí check `4`** (GREEN claimed, RED had `7`):
   `computeMod10('1590123412345') = 4`. CONFIRMED correct. The old RED vector `15901234123457` is wrong.

3. **GLN prefix `950110110700` ‚Üí check `7`** (GREEN claimed):
   `computeMod10('950110110700') = 7`. CONFIRMED correct. RED used a 10-digit prefix (`9501101107`) which was also the wrong input length for a GLN-12 prefix.

4. **GTIN-13 published reference `5901234123457`**: `computeMod10('590123412345') = 7`. CONFIRMED valid=true. AC1 passes.

Remaining corrections 5‚Äì7 are logical (not arithmetic): parseGTIN must accept both 13 and 14 digits, so a 13-digit input is never a length error and a 14-digit input with wrong check digit gives `check_digit_mismatch` not `length`. Both are correct per spec.
