### Top 5 must-fix (data-integrity / money / safety)

These are the gaps where a single normal user action causes irreversible, silent corruption of inventory, money, or food-safety state. **Fix these first.**

1. **`G-FIN-01` — WO actual cost sums PLN + EUR + GBP into one number with zero FX normalization** (`wo-cost-math.ts:96-121`). Three default currencies (item-cost `PLN`, process `EUR`, labor `GBP`) are added into one `totalCost` that drives cost-per-kg and pricing. **Every WO cost figure in the system is currently meaningless.** *Money — highest blast radius.*

2. **`G-SHIP-03` + `G-XCUT-04` + `G-QA-04` — Expired / QA-held / allergen-restricted stock ships to customers.** Shipping's allocate→pick→pack→ship chain re-checks nothing it filtered at allocation time: expired LPs (`so-actions.ts:651-662`), LPs put on hold *after* allocation (`ship-actions.ts:216`, `pack-actions.ts:310-336`), and customer "refuses-allergen" restrictions (`customer_allergen_restrictions` — *read by no code*). **Three independent regulatory/food-safety hard-stops, all absent in the ship path.** *Safety + recall liability.*

3. **`G-SHIP-04` — License-plate on-hand quantity is never decremented on ship** (`ship-actions.ts:216-245`). Ship flips status to `'shipped'` but never reduces `license_plates.quantity`; the `warehouse.lp.shipped` outbox event has **no consumer**. `cancelShipment` then re-exposes the full original quantity as available → physically-shipped goods become re-allocatable phantom stock. *Inventory integrity.*

4. **`G-PLAN-01` + `G-PLAN-02` — Desktop "Mark received" creates zero stock yet poisons MRP** (`actions.ts:801-856`). The PO Receive button flips status with no GRN/LP/stock created; MRP then either silently vanishes the qty from both supply and on-hand, or (partial) treats phantom inbound as real and suppresses a needed reorder. The goods also become **un-bookable** afterward (scanner excludes `received`). *Inventory + procurement ledger.*

5. **`G-PROD-01` + `G-PROD-03` — Disassembly and waste never draw down the source LP** (`register-disassembly-output.ts:309-444`, `record-waste.ts:144-193`). Breaking down a carcass spawns full-quantity co-product LPs while the input pallet stays 100% `available`; recording waste against an LP never reduces it. **Both double-count physical stock**, and disassembly can be re-run infinitely on the same pallet. *Inventory integrity — food traceability red flag.*

**Cross-cutting pattern:** the single most repeated root cause is **"validated once at allocation/creation, never re-asserted at the committing write."** Expiry, QA-hold, UoM, WO-state and reservation all leak through this seam. A shared `assertLpShippable(ctx, lpId, { asOf })` helper (mirroring the existing `assertLpConsumableForProduction`) would close ~8 of these in one slice.

## 3. REJECTED / ALREADY-GUARDED (so the owner sees coverage, not just holes)

These were investigated and the guard already exists — **do not re-open**:

1. **Scanner pick ignores reservations (WO-A's material picked into WO-B)** — FALSE. The committed quantity (`reserved_qty`) is protected at the **consume** layer: both consume paths require `quantity − qty >= reserved_qty`, and pick stages `available_qty` (= `quantity − reserved_qty`), so a fully-reserved LP stages zero. Plus DB CHECK `reserved_qty <= quantity` + `v_inventory_available` exclude the reserved portion. *(Residual soft-allocation nuance on partial reservations, not the claimed harm.)*
2. **`createTransferOrder` allows `from_warehouse == to_warehouse`** — FALSE. DB CHECK `transfer_orders_distinct_warehouses_check` (mig 263) is live + validated; an equal-warehouse row can never be persisted, so ship/receive can't process one either.
3. **Allocation `reserved_qty` can exceed on-hand** — FALSE. `license_plates_reserved_qty_le_quantity_check` (mig 191) is live + enforced (empirically rejected `reserved_qty = quantity+1` with 23514).
4. **Active CCP with both critical limits NULL silently passes every reading** — FALSE. `haccp_ccps_active_limit_required_check` (mig 321) is live + validated; an active CCP must carry ≥1 limit, and `recordMonitoring` only evaluates active CCPs.
5. **Factory release has no allergen-audit gate (stale cascade at release)** — FALSE. The published allergen set (`product.allergens`/`may_contain`) is trigger-maintained in the same txn as the recipe lock (mig 222 `fa_allergen_set_refresh_on_product_edit`); structurally can't be stale relative to the locked recipe.
6. **`materializeNpdBom` selects latest-numbered locked version, ignoring a newer DRAFT** — FALSE. The "locked v1 + newer draft v2" state is unreachable: `createFormulationDraft` short-circuits when `current_version_id` is set, there is no fork/unlock action, and the state-transition trigger forbids editing a locked version. Live data confirms 0 multi-version formulations.
7. **`createActiveNpdBom` defaults yield 100% / never validates range** — FALSE. `bom_headers.yield_pct CHECK (>0 AND <=100)` (mig 090) is live; out-of-range inserts throw. Source `target_yield_pct` is also bounded on `formulation_versions` (mig 093). *(Minor UX nit: a 0% formulation surfaces as a raw error, not a friendly blocker.)*
8. **WO actual cost `max(cost_per_kg)` per group picks highest of distinct components** — FALSE. Group key `coalesce(item_code, component_id)` + `(org_id, item_code)` unique constraint + org-scoped join means each group resolves to exactly one item/one cost; `max()` over one distinct value is lossless. *(Residual cosmetic: unmatched `component_id` valued at 0 with no warning — opposite of the claimed harm.)*
