# Bulk / Excel-Friendly Import — Proposal for MonoPilot Kira

## 1. What already exists (do NOT re-build)

MonoPilot already ships a real, Supabase-backed bulk-import capability — the **Wave E-IO engine** — plus several one-off importers. The proposal below extends this, it does not reinvent it.

### Stubs / disabled (must be finished or ignored, NOT re-proposed)
- **Settings master-data hub drawer** (`/settings/import-export`): upload + header-parse + field-map UI all exist, but **`[Run import]` is stubbed** (closes drawer, no action) and **`[Download template]` is a dead button**. The map-fields heuristic is reusable; the commit path is missing.
- `actions/import-export/import.ts` — `IMPORT_FEATURE_DISABLED=true`, only target `authorization_policies`, no UI.
- `actions/import-export/export.ts` — `EXPORT_FEATURE_DISABLED=true`, no UI.

### The reusable foundation (this is what every new importer plugs into)
- **`parse-entity-csv.ts`** — the generalisation: `EntityCsvSpec<TRow>` interface + `parseEntityCsv()`, `buildEntityTemplateCsv()`, `groupEntityRows()`, `buildEntityErrorReportCsv()`. **A new importer only declares a spec object.** (`parse-po-csv.ts` is the older byte-identical PO-specific copy — do not extend it.)
- **`entity-import-wizard.client.tsx`** — `EntityImportWizard<TRow,TCreated>`: the generic 4-step React wizard taking `spec` + `previewColumns` + `validateAction` + `commitAction`.
- **`entity-import-card.client.tsx`** — `EntityImportCard`: hub card with `[Download template]` + `[Import file]`, zero entity-specific logic.
- **`to-spec.ts` / `wo-spec.ts`** — worked `EntityCsvSpec` examples (coerce / cells / groupKey pattern) — copy these to declare a new spec.
- **`lib/shared/download.ts`** — `csvCell`, `toCsv`, `downloadCsv`, `downloadBlob`, `isoDateStamp`, `fileSafe`: canonical client-side CSV serialisation + browser download (used in 40+ files).
- **`parse-items-csv.ts`** — deeper **diff-aware** parser (`splitCsvLine`, `ImportDiffRow`, `REQUIRED_HEADERS`) — the model for upsert/update flows.

**Gap to close for "Excel-friendly":** everything today is **CSV-only**. The owner's customers live in **.xlsx**. The single biggest engine addition is a **.xlsx reader** (one library, one adapter that emits the same row arrays the existing parsers expect) plus a **"reference" sheet in templates** documenting enums — neither exists yet.

## 3. Proposed reusable Excel-import UX (ONE shared framework)

A single `<BulkImportModal>` driven entirely by an `EntityImportSpec` — never bespoke per entity. It is the **existing `EntityImportWizard` promoted to a modal + given an .xlsx reader**.

### Entry points
- A small **`[Import]` button sits next to every `+ New X`** on the entity's list page (e.g. `+ New Supplier` → `Import suppliers`).
- The **`/planning/import` hub stays** as the cross-module landing page; new entities appear there as additional `EntityImportCard`s so a power user can do them all in one place. (Master-data entities also surface in the existing `/settings/import-export` hub once its commit path is wired.)

### Modal flow (5 steps; idempotent commit)
1. **Start** — `[Download template]` (always present; generates .xlsx with a *Data* sheet + a *Reference* sheet, see §4) and a dropzone accepting **.xlsx and .csv**.
2. **Upload + auto-map** — file parsed client-side; headers fuzzy-matched to spec columns (reuse the master-data hub's auto-match heuristic). A **column-map table** shows `file header → target field`, with a per-column `<select>` to correct mismatches and an "unmapped" warning. Required columns missing → block with a clear list.
3. **Validate + row preview** — server `validateAction` runs the spec (type coerce, enum check, **FK-by-code resolution**, dedup check). A **per-row table** shows ✅ / ⚠ / ✖ with the **failing cell highlighted** and a human message ("supplier_code `ACME99` not found"). KPIs: N ok / N warn / N error. `[Download error report]` exports the failed rows + reason column (reuse `buildEntityErrorReportCsv`).
4. **Fix-in-place or re-upload** — invalid rows are **inline-editable** in the preview (then re-validate just-changed rows), or the user re-uploads. The WO-style **conversion/derived preview column** (`100 pcs → 50 kg`) shows where the spec declares `showConversion`.
5. **Commit** — choose **partial-commit (skip-invalid)** or **all-or-nothing**. Server `commitAction` **re-validates fail-closed** (like the items importer), runs inside `withOrgContext` + RLS, and is **idempotent** via a natural key (see §4 dedup). Result step: created/updated/skipped/failed counts + deep links to created records + a job ledger row (`import_export_jobs`, `kind='import'`) so it shows in the Settings hub.

### Why a modal *and* the hub
The owner asked for the import **next to the single-create flow** so Excel users stay in context — that is the modal. The hub is the bulk-onboarding console. Both render the **same `BulkImportModal`** from the **same spec** — zero duplication.

## 4. Template design rules (apply to EVERY entity)

1. **Human CODES, never UUIDs.** Every FK column is a code resolved server-side: `supplier_code→supplier_id`, `item_code→item_id`, `warehouse_code→warehouse_id`, `location_code→location_id`, `line_code→production_line_id`, `customer_code→customer_id`, `carrier_code→carrier_id`, `site_code→site_id`, `from_unit_code/to_unit_code→unit_of_measure.id`. **This explicitly fixes the two known UUID-in-CSV gaps** (`importLocationCsvAction` wants `warehouseId` UUID; reorder/line/shift/printer actions take UUIDs). The resolution map is built once per import (one `IN (...)` query per code domain), and unknown codes become a row-level error, never a silent null.
2. **Two-sheet .xlsx template.** *Sheet 1 "Data"* = the header row + one example row (greyed). *Sheet 2 "Reference"* = every enum/dropdown column with its valid values, and the valid `*_code` lists (suppliers, items, warehouses, uoms…) pulled live from the org so the user can VLOOKUP. CSV download keeps a `# valid values:` comment block for enums.
3. **Required vs optional, explicit.** Required headers are marked with a trailing `*` in the template and listed in the modal copy; optional columns may be omitted entirely. Missing required column = hard block at step 2.
4. **Enum columns documented + validated.** Statuses, types, categories, modes, etc. are listed in the Reference sheet and rejected server-side if out of range. Examples: supplier `status` ∈ active|inactive|blocked; item `item_type` ∈ fg|wip|rm|rm_supplier_specs; carrier/lane `mode`; `cost_basis`; quality `parameter_type`; CCP `hazard_type`.
5. **Date & number formats.** Dates = **`YYYY-MM-DD`** (ISO, validated). Numbers = plain decimal, **`.` decimal separator**, currency 3-char ISO (`EUR` default). Reject Excel serial-date numbers and thousands separators with a clear message.
6. **One example row, always.** Reuse the `PO_IMPORT_EXAMPLE_ROW` pattern — every spec ships a realistic example so the user sees the exact shape.
7. **Idempotency / dedup key per entity.** Each spec declares its natural key (e.g. supplier `code`, customer `code`, item-cost `item_code+effective_from`, reorder `item_code`, temp-range `item_code`, UoM `code`). Re-importing the same key **updates** (or **skips**, per a row's behaviour radio) instead of duplicating — surfaced as the diff badges already built in `parse-items-csv.ts`.

## 6. Build approach + phasing

### Engine decision: **extend the E-IO engine, don't fork a new package**
The `EntityCsvSpec` + `EntityImportWizard` + `EntityImportCard` trio is already the generic engine. Promote and harden it in place rather than create a `packages/import`:
- **Add an .xlsx adapter** — one library (`xlsx`/SheetJS) behind a thin `readSpreadsheet(file): { headers, rows }` function that emits the **same row arrays** `parseEntityCsv()` already consumes. CSV path unchanged. This is the one genuinely new dependency.
- **Add server-side FK-code resolution to the spec** — extend `EntityCsvSpec` with a `resolvers` map (code-domain → loader); the validate/commit actions build the lookup once and attach `*_id` to each row. Reuse the `import-po.ts` `loadSuppliers`/`loadActiveItems` pattern.
- **Add the two-sheet template builder** — extend `buildEntityTemplateCsv()` with `buildEntityTemplateXlsx()` that writes Data + Reference sheets from the spec's enum/code declarations.
- **Add upsert/diff** for master data — borrow `ImportDiffRow` from `parse-items-csv.ts` so re-imports update rather than duplicate.
- **Promote the wizard into `<BulkImportModal>`** so it can mount next to `+ New X` and inside both hubs.
- **Wire the stubbed master-data hub** (`/settings/import-export`) to the same `commitAction` — its auto-map UI is reusable; only the dead `[Run import]` / `[Download template]` need connecting.
- **Consolidate** `parse-po-csv.ts` onto `parse-entity-csv.ts` (kill the duplicate `splitCsvLine`) so there's one parser.

### Phasing
**Phase 1 — master data (highest ROI, lowest risk; all upsert-by-code):**
Suppliers (1), Customers (2), Machines (8), UoM + conversions (9), Reorder thresholds (7), Product temp ranges (10), Item cost roll (6), and the **Location warehouse-code fix**. These are self-contained or single-code-FK — ideal first cut and they validate the .xlsx adapter + resolver + two-sheet template + diff/upsert on simple cases.
*Reusable bits proven here:* `readSpreadsheet`, `resolvers`, `buildEntityTemplateXlsx`, `ImportDiffRow`, `<BulkImportModal>`.

**Phase 1.5 — multi-line grouped master data:**
BOM header+lines (3), BOM line-into-draft (4), Quality spec+parameters (5). These exercise the **group-by-code** path (`groupEntityRows`, the PO grouping pattern) on master data before transactions.

**Phase 2 — transactional + governed-adjacent:**
Sales Orders (11, PO-importer clone), CCPs (12, upsert on `ccp_code`), Item nutrition (13), Supplier specs (14), Carriers + lanes (15, gate on E9), Site/Warehouse/Line/Shift (16), NPD formulation/packaging (17,18).

### Risks & guardrails
- **FK resolution** — unknown code = row-level error with the offending value, never a silent null/insert. Resolve in **one batched query per code domain**; for grouped imports resolve the header code once per group.
- **RLS / org-scoping** — every validate + commit runs under `withOrgContext`; resolvers query only the caller's org (RLS). The Reference sheet's live code lists are equally org-scoped. Honour Wave0 lock (`org_id`, `app.current_org_id()`).
- **Dedup / upsert semantics** — each spec declares a natural key; default behaviour is **update-on-conflict** with a per-row **skip** option, surfaced via the diff badges. Idempotent commit (re-running a file is a no-op for unchanged rows) via that key.
- **Partial-commit** — offer both all-or-nothing and skip-invalid (already in the PO wizard); commit **re-validates fail-closed** server-side so a stale preview can't push bad rows.
- **EXCLUDE e-signed / governed entities** — never bulk-import anything carrying a mandatory reason, e-signature, supervisor PIN, immutable audit ledger, real-time hold/breach logic, or auth side-effects (full list in §2 "EXCLUDED"). For these the single-create flow stays the only path by design.
- **RBAC** — reuse the existing per-entity write permission as the import gate, resolved server-side on modal open (mirrors `npd.planning.write`, `settings.reference.import`, `technical.items.create`).
- **.xlsx parsing edge cases** — Excel serial dates, locale decimal/thousands separators, multi-tab files: normalise in `readSpreadsheet`, reject ambiguous values with a clear per-cell message rather than guessing.
