## 1. Ground truth: bom_headers.product_id today

**Column shape (live DB, confirmed from `packages/db/__expected__/schema.sql` line 6973 + mig-090 original DDL):**

```
product_id   text   NULL
```

- Type: `text`; nullable; no NOT NULL constraint.
- Value stored: the FG's **product_code** (= `items.item_code` = `product_legacy.product_code`). This is a natural key, NOT a uuid.
- FK constraint: `bom_headers_product_id_fkey` — composite `(org_id, product_id)` references `public.product(org_id, product_code)` ON DELETE RESTRICT. After mig-359 `public.product` is a view, but this FK was declared against it; the live FK now resolves to the `product_legacy` skeleton anchor table (because Postgres resolves FK targets by OID, and after the `ALTER TABLE product RENAME TO product_legacy` the OID auto-followed — confirmed by mig-359:37 comment "The 16 FK satellites, the 3 dependent views, and the 4 triggers all auto-follow the rename").

Evidence:
- `packages/db/__expected__/schema.sql:6973` — column definition
- `packages/db/__expected__/schema.sql:17533` — FK constraint DDL
- `packages/db/migrations/090-shared-bom-ssot-npd-origin.sql:9` — original declaration
- `packages/db/schema/shared-bom.ts:36-65` — Drizzle schema (productId: text, foreignKey referencing product.orgId + product.productCode)
- `packages/db/migrations/359-product-as-items-view-cut.sql:8-9` — confirms FK satellites auto-follow rename

**Is there already an items.id available to writers?**
Yes — `items.item_code` = `product_code` is the join key. Every writer that has the product_code can resolve `items.id` via `select id from public.items where org_id = app.current_org_id() and item_code = $1`. The INSTEAD-OF INSERT trigger in mig-359 already does exactly this lookup.

## 5. Migration approach — Recommended PHASED cut (add item_id, dual-write, switch, drop)

### Why phased rather than single cut

A single cut (rename column in-place, update all 25 reader callsites atomically) would require:
- Deploying a new Vercel build with every SQL query rewritten simultaneously.
- Zero downtime only if the rename and the code change are deployed in a single atomic step — impossible across DB migration + Vercel deploy boundary.
- 25 reader callsites + 3 writer callsites + 2 DB functions + 1 clone trigger modified in one PR, with no ability to roll back the column rename without cascading failures.

The product→items merge (mig-359) used a PHASED approach (skeleton anchor, INSTEAD-OF triggers, keep old surface) for exactly this reason. P3-FK follows the same pattern.

### Recommended approach: four discrete migration steps (migs A, B, C, D)

**Mig A — Add `item_id uuid` column (no FK yet, nullable, no reader change)**

```sql
-- Mig A: add shadow column
ALTER TABLE public.bom_headers
  ADD COLUMN IF NOT EXISTS item_id uuid;

-- Backfill: resolve text product_id → items.id via item_code match
UPDATE public.bom_headers h
SET item_id = i.id
FROM public.items i
WHERE i.org_id = h.org_id
  AND i.item_code = h.product_id
  AND h.product_id IS NOT NULL
  AND h.item_id IS NULL;

-- Add the FK (deferred until after backfill to avoid locking)
ALTER TABLE public.bom_headers
  ADD CONSTRAINT bom_headers_item_id_fkey
  FOREIGN KEY (item_id) REFERENCES public.items(id)
  ON DELETE RESTRICT;

-- Completeness check (must return 0 before proceeding)
SELECT count(*) FROM public.bom_headers
WHERE product_id IS NOT NULL AND item_id IS NULL;
```

**Mig B — Dual-write trigger: keep item_id in sync with product_id on INSERT/UPDATE**

```sql
CREATE OR REPLACE FUNCTION public.bom_headers_sync_item_id()
RETURNS trigger LANGUAGE plpgsql
SET search_path TO 'public', 'pg_catalog'
AS $$
BEGIN
  IF NEW.product_id IS NOT NULL AND NEW.item_id IS NULL THEN
    SELECT i.id INTO NEW.item_id
    FROM public.items i
    WHERE i.org_id = NEW.org_id AND i.item_code = NEW.product_id
    LIMIT 1;
  END IF;
  RETURN NEW;
END;
$$;

CREATE TRIGGER bom_headers_sync_item_id_trg
  BEFORE INSERT OR UPDATE ON public.bom_headers
  FOR EACH ROW EXECUTE FUNCTION public.bom_headers_sync_item_id();
```

**Code Phase C — Rewrite readers and writers to use item_id (deploy separate PR per callsite group)**

Order of reader migration (low-risk first):
1. Internal utility readers (dashboard-kpis, cost, traceability, where-used, allergen cascade) — join `items i ON i.id = bh.item_id` instead of `product pr ON pr.product_code = bh.product_id`.
2. BOM list/detail queries (queries.ts, detail-page.ts, history.ts, diff-action.ts).
3. Cycle detection graph builders (workflow.ts, create-draft.ts, line-actions.ts) — these use product_id as a text KEY for cycle detection; switching to item_id (uuid) requires updating the graph node identity from code to uuid throughout the cycle-detection algorithm.
4. Writers (create-draft.ts, disassembly.ts, materialize-npd-bom.ts) — add items.id lookup, pass `item_id` column.
5. DB functions (get_fa_bom, build_shared_bom_from_npd in mig-099, clone trigger in mig-168) — replace with updated versions.

**Mig D — Drop product_id column and legacy sync trigger (after all code is on item_id)**

```sql
DROP TRIGGER IF EXISTS bom_headers_sync_item_id_trg ON public.bom_headers;
DROP FUNCTION IF EXISTS public.bom_headers_sync_item_id();
ALTER TABLE public.bom_headers DROP CONSTRAINT bom_headers_product_id_fkey;
ALTER TABLE public.bom_headers DROP COLUMN product_id;
-- (also drop the 4 partial indexes that reference product_id)
DROP INDEX IF EXISTS bom_headers_org_product_version_unique;
DROP INDEX IF EXISTS bom_headers_org_npd_project_version_unique;
DROP INDEX IF EXISTS bom_headers_org_product_idx;
DROP INDEX IF EXISTS bom_headers_active_version_idx;
-- Re-create equivalents on item_id
CREATE UNIQUE INDEX bom_headers_org_item_version_unique ON public.bom_headers (org_id, item_id, version) WHERE item_id IS NOT NULL;
CREATE UNIQUE INDEX bom_headers_active_version_idx ON public.bom_headers (org_id, item_id) WHERE status = 'active' AND item_id IS NOT NULL;
CREATE INDEX bom_headers_org_item_idx ON public.bom_headers (org_id, item_id, status, version DESC) WHERE item_id IS NOT NULL;
```

**Rollback plan:**

- Mig D is the point of no return. Gate: hold mig D until at least 72 hours of dual-write soak with zero errors logged.

## 6. Risk list

| Risk | Severity | Detail |
|---|---|---|
| **Orphan bom_headers rows** | HIGH | If any `bom_headers.product_id` has no matching `items` row (e.g., items was never inserted for an old FA product), backfill leaves `item_id = NULL` and the FK gate in mig D would block. Pre-check must find and fix these first. |
| **Cycle detection identity swap** | HIGH | The cycle detection algorithm in `create-draft.ts:117-131` and `workflow.ts:79-90` uses `product_id` (text code) as the graph node key. Switching to item_id (uuid) requires every graph node and component comparison to use uuid. A partial migration (reader code on item_id, cycle detection still on code) creates a window where cycle detection always passes (no match) — silent data corruption risk. |
| **Clone-on-write trigger (mig-168)** | MEDIUM | The state-machine clone trigger in mig-168 copies `product_id` verbatim. If mig D drops the column before the trigger is updated, the clone trigger errors on INSERT. The trigger must be rewritten in Phase C before mig D. |
| **get_fa_bom DB function (mig-099)** | MEDIUM | The `get_fa_bom` and `build_shared_bom_from_npd` functions in mig-099 use `header.product_id = v_product_id` (text). These are SQL DB functions that must be replaced with a new migration (not app-layer code). If dropped prematurely, the FA BOM tab breaks. |
| **bom_not_orphaned CHECK constraint** | LOW | The CHECK `product_id IS NOT NULL OR npd_project_id IS NOT NULL OR fa_code IS NOT NULL` must be updated to reference `item_id` when product_id is dropped. Easy to do in mig D but easy to forget. |
| **Unique index on product_id** | LOW | Four partial indexes reference `product_id`. Mig D must drop and recreate them on `item_id`. Missing this causes postgres to plan-fail on index scans for those shapes. |
| **Drizzle schema drift** | LOW | `packages/db/schema/shared-bom.ts` defines `productId: text(...)` with a foreignKey to `product`. After the column rename the Drizzle schema must be updated in Phase C (before mig D), or `pnpm db:push` would fail the drift gate. |

## 8. P5 scope: drop product_legacy — which of the 16 satellites must first move to items.id

`product_legacy` cannot be dropped until ALL 16 FK satellites are repointed. The three with RESTRICT/NO ACTION are the hardest (they will block DELETE of any product_legacy row):

| Priority | Table | Action needed before P5 |
|---|---|---|
| Critical (blocks P3-FK-D) | `bom_headers` (#1) | Done by P3-FK itself |
| Critical (RESTRICT) | `factory_release_status` (#7) | Repoint `product_code text` → `fg_item_id uuid references items(id)` |
| Critical (RESTRICT) | `npd_legacy_closeout` (#9) | Repoint `fg_product_code text` → `item_id uuid references items(id)` |
| Critical (NO ACTION) | `formulations` (#8) | Repoint `product_code text` → `item_id uuid references items(id)` |
| Critical (NO ACTION) | `npd_projects` (#10) | Repoint `product_code text` → `item_id uuid references items(id)` |
| Lower (CASCADE — product_legacy row delete will cascade) | #2,3,4,5,6,11,12,13,14,15,16 | Repoint opportunistically; cascade semantics mean P5 can succeed even if they still reference product_legacy rows that get deleted |

Practically: P5 requires at minimum repointing the 4 RESTRICT/NO-ACTION satellites (#7-10) before product_legacy rows for any used product can be deleted. The CASCADE satellites (#2,3,4,5,6,11-16) could be left if acceptable to cascade-delete their data — but that would be destructive and is almost certainly not acceptable for `fa_allergen_overrides`, `formulations`, `risks`, etc.

**Realistic P5 timeline:** after all 16 satellites are repointed (a separate wave of 16 migrations), product_legacy can be dropped cleanly.

## 9. Recommendation

**Do P3-FK as PHASED (4 steps: mig A, mig B, code-Phase-C per callsite group, mig D).**

- Mig A (backfill) + mig B (dual-write trigger) can ship in the next migration slot with zero code change and zero reader impact.
- Code Phase C is the high-effort part: 25 reader callsites, 3 writer callsites, 2 DB functions, 1 clone trigger. Recommend sub-grouping into at least 3 PRs (internal readers / BOM detail+list / cycle detection + writers + DB functions).
- Mig D (drop product_id) is the irreversible step. Gate it behind 72h soak + verified completeness check.

**Biggest single risk: cycle detection identity swap (HIGH).** The graph nodes currently use `product_id` (text code) as their identity key in `create-draft.ts` and `workflow.ts`. Switching to uuid requires consistent replacement across the `parent`/`component` graph shape, the `detectCycle` call parameters, and the `buildGraph` edge construction. A partial migration leaves a silent cycle-detection bypass. This callsite group must be migrated atomically (all three files together in one PR): `create-draft.ts`, `workflow.ts`, `line-actions.ts`.
