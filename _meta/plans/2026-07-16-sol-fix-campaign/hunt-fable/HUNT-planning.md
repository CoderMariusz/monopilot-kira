## Details

### NEW-P1 ‚Äî capacity/shift-infeasible WO crashes the entire scheduler run (P1)
`resolvePlannedStart` (sequence-solver.ts:285-328) throws `SequenceCapacityInfeasibleError` when:
- capacity is finite and the run can never fit a single UTC day's remaining budget (loop hits `MAX_CAPACITY_PLACEMENT_ATTEMPTS = 400`, line 299/327); or
- shift windows are configured and `nextShiftStart` returns null because the run is longer than any single contiguous shift window (line 300-303 ‚Üí `break` ‚Üí falls through to line 327).

`placeSequencedWorkOrder` (388-433) and `sequenceWorkOrders` (435-527) do NOT wrap this call in try/catch. The only catch is the outer server-action boundary at scheduler-actions.ts:1103, which maps ANY throw to `{ ok:false, error:'persistence_failed' }`. Result: one pathological WO (long routing_duration_ms/process_duration_ms, or a WO longer than the line's shift blocks) discards the schedule for every other WO in that run.

Default reachability confirmed: `solverConfigFromRow(null)` ‚Üí `cloneDefaultSolverConfig()` ‚Üí `capacityHoursPerDay = 16` (finite). So a line with no scheduler_config row already has a 16h/day cap; a >16h WO triggers it. Shift path is even easier (e.g. 10h WO vs 8h shifts).

Fix direction (matches existing pattern): catch `SequenceCapacityInfeasibleError` in the per-WO placement, push the WO to `omitted` with a new reason `no_feasible_capacity`, continue the run ‚Äî exactly how infeasible-changeover WOs are already deferred/omitted (line 490-493, 519-524).

### NEW-P2 ‚Äî reserved capacity excludes changeover + span (P2)
`placeSequencedWorkOrder` computes `earliestStart = max(now, plannedEndByLine + changeoverCost*60000)` (line 411) ‚Äî so changeover pushes the start later ‚Äî but then calls `resolvePlannedStart`/`reserveCapacity` with only `runDuration` (line 413, 316-317). `reserveCapacity` (250-261) adds just the run's per-day overlap to `dayUsageMs`. The changeover gap and any idle span are never charged. Two 8h runs separated by a 4h changeover both "fit" a 16h/day line even though they occupy 0‚Äì20h of wall clock. Over time this schedules lines beyond real capacity. Fix: charge `changeoverCost` (and/or reserve the full `[earliestStart, plannedEnd]` span) against `dayUsageMs`, not just `runDuration`.

### NEW-P3 ‚Äî MRP SQL horizon wider than phased bucket grid (P3)
`runMrp` computes `horizonEnd = planningHorizonEnd(today, horizonWeeks) = today + weeks*7` (mrp.ts:219) and uses it in every demand/supply WHERE (`... <= $3::date`, lines 258, 325, 378, 399). But `computeMrpPhased` builds buckets from `buildMrpBucketDates(today, weeks)` starting at the Monday of today's ISO week, so the grid ends at `bucketHorizonEnd = firstMonday + weeks*7 - 1` (mrp-buckets.ts:59-62), which is 1‚Äì7 days EARLIER than the SQL `horizonEnd` whenever today is not a Monday. Rows dated in that trailing sliver satisfy the SQL filter, reach `applyTimed`/`applyForecast`, and `dateToBucketIndex` returns `OUT_OF_HORIZON_BUCKET_INDEX` (mrp-buckets.ts:84-85) ‚Üí the loops `continue` (mrp-compute.ts:682, 702) and the qty is dropped from every bucket and from the summary (summary is rolled up from bucket accumulators). Net: up to a week of near-horizon demand/supply is silently invisible. Persistence uses `bucketHorizonEnd` consistently (mrp.ts:528), so only the SQL fetch is mis-bounded. Fix: bound the SQL WHERE by `bucketHorizonEnd(buildMrpBucketDates(today, weeks))` instead of `today + weeks*7` (or align the bucket grid start to `today`).

### NEW-P4 ‚Äî self-transfer / warehouse-less TO accepted at create (P3)
`TransferOrderCreateInput` (procurement-shared.ts:110-121) validates lines but never enforces `fromWarehouseId !== toWarehouseId`; both fields are `.optional()`. `createTransferOrderCore` inserts the header verbatim (create-transfer-order-core.ts:145-163). A TO whose source and destination warehouse are identical becomes a no-op transfer that still emits pick/ship stock_moves and re-materializes a duplicate dest LP in the same warehouse on receive ‚Äî matter conservation passes (`assertTransferOrderMatterConserved`), so nothing downstream rejects it. Add a `.refine(from !== to)` (and require both present for non-draft) at the schema, mirroring the same-location guard shipped for C101.
