## Systemic patterns (THE KEY SECTION)
1. **raw `<select>` instead of shadcn `<Select>`** — affects ≥5 screens (roles, tenant/depts, import-export, tenant/migrations, schema/new; also seen in drift/dlq comments claiming none). Root cause: these selects sit in Server Components or in places where pulling in the client `<Select>` was avoided; the team left native `<select>`. This is a named **hard red-line** in `MON-t3-ui`. **Fix belongs in a SKILL clarification** (MON-t3-ui: give the canonical RSC→Select pattern, e.g. a thin client `<SelectField>` wrapper) THEN per-screen execution to swap. ~5–8 screens.
2. **Generic placeholder stub for screens that have (or should have) a real prototype** — affects 7 screens via `SettingsRouteStub` (boms, devices, gallery, labels, products, shifts, ship-override-reasons), 2 of which (shifts, ship-override-reasons) map to real prototype anchors. Root cause: deferred/unbuilt screens still surfaced in nav. **Per-screen EXECUTION** (build the screen or remove the nav entry); track in STATUS Bucket C.
3. **Visual density drift toward heavy Tailwind cards** (`rounded-3xl/p-8/shadow-sm`, `rounded-2xl/p-4`) vs prototype's compact admin density — affects all stub + launcher pages (onboarding, 7 stubs) and is the default look of the `SettingsRouteStub`. Root cause: stub component authored without prototype density tokens. **SKILL/shared-component fix** (retire/restyle SettingsRouteStub) — but low priority since these are placeholders.
4. **permission-denied not surfaced as a distinct UI state on read-gated screens** — confirmed on security (maps denied→ready); needs spot-check on features/notifications/promotions/modules (grep shows no `permission-denied`/`canManage` token in those page.tsx). Root cause: pages return data with default values instead of a denied banner. **Per-screen EXECUTION**, but worth a SKILL note that read-permission-denied must render the amber denied banner pattern used by users/page.
