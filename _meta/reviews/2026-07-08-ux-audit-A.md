### [A-01] Editing packaging components silently removes the catalog item FK
**File:** apps/web/app/[locale]/(app)/(npd)/pipeline/[projectId]/packaging/_components/packaging-component-modal.tsx:56
**Class:** Modal-parity / prototype drift
**Impact:** A packaging component created from the item picker is shown as linked to the item master, but the edit modal does not receive the existing `item_id`; `rowToForm()` resets `itemId` to `null` and the submit payload sends that null. The server update then writes `item_id = $13::uuid`, so changing any editable field on a linked component clears the catalog link. Users lose item-driven supplier/price/spec traceability just by editing packaging metadata.
**Fix sketch:** Include `item_id`/`item_code` in the packaging loader row and `PackagingComponentRow`, seed the edit form from those values, and only clear `itemId` when the user explicitly clicks the clear-item control.

### [A-02] Packaging supplier is typed as free text instead of selected from suppliers
**File:** apps/web/app/[locale]/(app)/(npd)/pipeline/[projectId]/packaging/_components/packaging-component-modal.tsx:269
**Class:** Dropdown-should-be-FK
**Impact:** The add/edit packaging modal lets users type any `supplierCode`, and the write action persists `supplier_code` directly. A typo or obsolete code still lands in `packaging_components`, so packaging cost/supplier attribution can drift from the real supplier master and cannot reliably drive procurement handoff.
**Fix sketch:** Load active suppliers for the org and replace the supplier `<Input>` with a supplier picker/select that submits `supplierId` or a validated canonical supplier code. Keep the picker prefilled from the selected packaging item when available.

### [A-03] ECO list hides all rows after the first 100
**File:** apps/web/app/[locale]/(app)/(modules)/technical/eco/_actions/page-data.ts:63
**Class:** List truncation
**Impact:** `loadEcoPage()` hard-caps `technical_change_orders` at `LIST_LIMIT = 100`, while the page table renders only those rows and exposes no pagination or load-more control. The status counts are computed from the full table, so users can see that more ECOs exist but cannot open or act on any row beyond the first 100 for the selected status.
**Fix sketch:** Add real server-side pagination/search to `loadEcoPage()` and render pagination controls, or remove the cap only if the expected row count is safely bounded.

### [A-04] Production changeover register hides all rows after the first 100
**File:** apps/web/app/[locale]/(app)/(modules)/production/changeovers/page.tsx:110
**Class:** List truncation
**Impact:** The changeover page calls `listChangeovers({ limit: 100 })`, and `listChangeovers()` enforces that limit against `changeover_events`. The client status chips then filter only that first page in memory, with no pagination or load-more UI. Older pending or first-signed allergen changeovers can become invisible, leaving operators unable to review/sign them from the register.
**Fix sketch:** Thread status/page parameters into `listChangeovers()`, fetch the active filter server-side, and render pagination or load-more controls. Avoid client-only filtering over a capped server result.
