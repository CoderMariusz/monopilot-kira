## D1 — Decision Rationale Source 2026-04-30: 10 system roles, 4 customer-facing role categories, role_categories

- Decision: Keep **10 system roles** for DB/RBAC permission resolution: `owner`, `admin`, `npd_manager`, `module_admin`, `planner`, `production_lead`, `quality_lead`, `warehouse_operator`, `auditor`, `viewer`. Add/consume a read-only `role_categories` mapping so the Users UI can display **4** customer-facing categories: Admin, Manager, Operator, Viewer.
- Rationale: This preserves fine-grained system permissions while matching the simpler customer-facing UX model. The 4-category layer is presentational only and must not replace the underlying role codes.
- Date: 2026-04-30.
- Source: [02-SETTINGS PRD v3.5 §3](../../docs/prd/02-SETTINGS-PRD.md#3-personas-rbac-overview), gap-backlog D1.

## D2 — Decision Rationale Source 2026-04-30: flat permission, checkbox UI, no module_permission_levels

- Decision: Keep the **flat permission** model (`module_code.action.scope` dot strings) and implement the Permissions UI as checkbox-per-permission grouped by module. Reject the 4-level prototype matrix as the canonical data model; do not create a `module_permission_levels` derived view/table for this wave.
- Rationale: `permissions.enum.ts` remains the single source of truth. A checkbox UI can present the flat permission set without inventing a second permission abstraction that would drift from enforcement.
- Date: 2026-04-30.
- Source: [02-SETTINGS PRD v3.5 §3](../../docs/prd/02-SETTINGS-PRD.md#3-personas-rbac-overview), gap-backlog D2.

## D3 — Decision Rationale Source 2026-04-30: SET-001 through SET-006, no launcher

- Decision: Use SET-001 through SET-006 as the onboarding analytics/Playwright contract and drop the prior standalone **launcher** screen code. SET-001 is Org Profile Step and SET-006 is Completion Celebration.
- Rationale: The UX contract has six steps and no separate launcher. The wizard auto-mounts on `/onboarding` via the redirect-while-incomplete guard, so downstream tests/tasks must target SET-001..SET-006.
- Date: 2026-04-30.
- Source: [02-SETTINGS PRD v3.5 §14.4](../../docs/prd/02-SETTINGS-PRD.md#14.4-ui-surfaces-s-u1-per-gap-backlog-2026-04-30-renumbered-set-001..set-006-to-match-ux-dropped-launcher-per-d3), gap-backlog D3.

## D4 — Decision Rationale Source 2026-04-30: Sensory stage status

- Decision: Treat **Sensory** as build/retained NPD stage-screen scope unless a later Technical-owned sensory read model explicitly absorbs it; do not classify Sensory as a deprecated legacy stage by default.
- Rationale: The gap-backlog identified Sensory as the only related stage screen without a legacy banner. Keeping it prevents accidental deletion of sensory evidence while allowing Technical policy/read-model integration to own the final data source.
- Date: 2026-04-30.
- Source: [_meta/plans/2026-04-30 UX↔PRD gap-backlog D4](../plans/2026-04-30-ux-prd-plan-gap-backlog.md).

## D5 — Decision Rationale Source 2026-04-30: Kanban, Table, Split view modes

- Decision: Lock the NPD pipeline modes to **Kanban**, **Table**, and **Split**. URL/view contracts should use the three-mode model, not a two-mode Kanban/List-only model.
- Rationale: The prototype exposes three modes, and Split is needed for the left-table/right-detail workflow. Downstream work should align PRD/task ACs to Kanban/Table/Split rather than carrying the older two-view wording.
- Date: 2026-04-30.
- Source: [_meta/plans/2026-04-30 UX↔PRD gap-backlog D5](../plans/2026-04-30-ux-prd-plan-gap-backlog.md).

## D6 — Decision Rationale Source 2026-04-30: 01NPDg, 01NPDh, 01NPDi naming

- Decision: Resolve the g/h/i naming collision as: `01NPDg` = Recipe + Formulation editor; `01NPDh` = Stage screens for Nutrition + Costing + Sensory; `01NPDi` = Approval gate extension + Risk Register + Compliance Docs.
- Rationale: This removes the collision where multiple backlog slices proposed `01NPDg` for different work and gives each new NPD workstream a stable downstream task prefix.
- Date: 2026-04-30.
- Source: [_meta/plans/2026-04-30 UX↔PRD gap-backlog D6](../plans/2026-04-30-ux-prd-plan-gap-backlog.md).

## D7 — Decision Rationale Source 2026-04-30: WebAuthn Phase 3 disabled Coming Phase 3

- Decision: Keep **WebAuthn** deferred to **Phase 3**. If the security UI shows the WebAuthn option, render it `disabled` and label/tooltip it as **Coming Phase 3**.
- Rationale: This preserves the UX affordance without implying Phase 1 support or enrollability. TOTP/SMS policy remains configurable; WebAuthn must not be enabled until the Phase 3 implementation lands.
- Date: 2026-04-30.
- Source: [02-SETTINGS PRD v3.5 §14.1](../../docs/prd/02-SETTINGS-PRD.md#14.1-security-policies-per-5.7-s-u5-per-gap-backlog-2026-04-30), gap-backlog D7.
